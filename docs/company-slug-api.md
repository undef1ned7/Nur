# Company Slug API (редактирование и проверка уникальности)

Спецификация для редактируемого **slug компании** во вкладке «Онлайн» страницы `/crm/set`.
Slug используется в публичных путях (`app.nurcrm.kg/{slug}`, например
`/service/{slug}/booking`, `/catalog/{slug}`).

- Базовый URL: `https://app.nurcrm.kg/api`
- Авторизация: `Authorization: Bearer <accessToken>`
- Фронт-валидация — в `src/utils/slug.js`; **backend обязан повторить полную валидацию** и
  никогда не доверять только фронту.

---

## Формат slug

- Разрешено: строчные латинские буквы `a-z`, цифры `0-9`, дефис `-`.
- Запрещено: пробелы, кириллица, спецсимволы, точки, подчёркивания, двойные дефисы `--`,
  дефис в начале/в конце.
- Длина: **3..50** символов.
- Регэксп: `^[a-z0-9]+(?:-[a-z0-9]+)*$`
- Сервер нормализует вход (lowercase, trim) перед проверкой.

---

## 1. Async-проверка доступности — `GET /users/company/check-slug/?slug=<value>`

Позволяет фронту проверять slug **без сохранения** (дебаунс при вводе).

**Response 200:**
```json
{ "available": true }
```
```json
{ "available": false, "message": "Slug already exists" }
```

- Если формат невалиден — `available: false` + понятное `message`.
- Текущий slug компании-запросившего считается доступным для неё самой (`available: true`).
- Эндпоинт лёгкий: один индексированный `EXISTS`-запрос.

> Примечание: в ТЗ упомянут также вид `GET /api/company/check-slug?slug=`. Здесь закреплён путь
> `/users/company/check-slug/` для единообразия с остальными company-эндпоинтами
> (`/users/company/`, `/users/settings/company/`). Если бэкенд выберет `/company/check-slug`,
> фронту достаточно поменять путь в `src/api/auth.js` (`checkSlug`).

---

## 2. Сохранение slug — `PATCH /users/settings/company/`

Тело (частичное обновление компании):
```json
{ "slug": "my-company" }
```

**Серверная логика (обязательно):**
1. Нормализовать и провалидировать формат (тот же регэксп/длина). При нарушении — `400` с
   полем `slug` и сообщением.
2. Проверить уникальность по **нижнему регистру** среди всех компаний.
   - Если slug занят **другой** компанией → `409 Conflict`
     `{ "slug": ["Slug already exists"] }`.
   - Если это текущий slug компании — успех (no-op).
3. Сохранить в транзакции; полагаться на **уникальный индекс** в БД (последняя линия защиты от гонок).

**Response 200:** обновлённый объект компании (с новым `slug`).

---

## Надёжность и производительность

- **Уникальный индекс** на `lower(slug)` (например, `CREATE UNIQUE INDEX ... ON company (lower(slug))`)
  — гарантирует уникальность даже при гонке двух запросов (ловим `IntegrityError` → `409`).
- Проверка `check-slug` и финальная проверка при `PATCH` идут по одному индексу → быстро.
- Нормализация на сервере исключает обход фронт-валидации (curl/мобильный клиент).
- Опционально: запрет смены slug чаще, чем раз в N дней (для стабильности публичных ссылок) —
  на усмотрение бизнес-логики.

## Frontend (реализовано)

- `src/utils/slug.js` — `normalizeSlug`, `finalizeSlug`, `validateSlug`.
- `src/api/auth.js` — `checkSlug(slug)`.
- `Settings.jsx` (вкладка «Онлайн», карточка барбершопа): редактируемое поле slug с живой
  нормализацией, показом ошибок/доступности, дебаунс-проверкой и кнопкой «Сохранить»
  (`updateUserCompanyName({ slug })` → `getCompany()`). Доступно владельцу/админу.
