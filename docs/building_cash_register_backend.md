# Касса (Building Cash Register) — Backend-спецификация

Базовый префикс: `/api/building/` (уточняется по проекту).

Документ описывает:
- как при **продаже квартиры** и **оформлении рассрочки** создаются заявки в кассу;
- как касса **одобряет/отклоняет** заявки, прикрепляет файлы и сохраняет выплаты;
- как любые **движения по кассе** (поступления/расходы) могут иметь прикреплённые файлы.

Спецификация по зарплате и авансам: `building_payroll_backend.md`, `building_payroll_advance_requests_api.md`.

---

## 1. Общие понятия

- **CashFlow** — движение по кассе (приход/расход), базовая запись:
  - `id`
  - `cashbox` — uuid кассы
  - `shift` — uuid смены (опционально)
  - `amount` — сумма
  - `type` — `income` | `expense`
  - `status` — `draft` | `approved` | `cancelled`
  - `source_business_operation_id` — связь с бизнес-операцией (продажа, аванс, возврат и т.п.)
  - `created_at`, `approved_at`, `cancelled_at`
  - `comment`
- **Заявка на кассу** — сущность, которая создаётся бизнес-модулем (продажа квартиры, аванс по ЗП, возврат и т.д.) и должна быть **одобрена кассой**:
  - типы заявок: `apartment_sale`, `installment_initial_payment`, `installment_payment`, другие по мере необходимости;
  - после одобрения создаётся CashFlow и, при необходимости, изменяются статусы в бизнес-модуле (продажа/рассрочка).

Во всех операциях, описанных ниже, должна быть возможность **прикреплять файлы** к заявке и/или к самому движению по кассе (CashFlow).

## 2. Заявка на кассу при продаже квартиры

### 2.1 Создание заявки при продаже

При оформлении **продажи квартиры** (создание/подписание договора продажи) backend **не должен сразу создавать движение по кассе**, а должен сначала создать **заявку на кассу**:

- **`POST /cash-register/requests/`** (внутренний вызов из модуля продаж)  
  Тип заявки: `apartment_sale`.

Пример тела (упрощённо):

```json
{
  "request_type": "apartment_sale",
  "treaty": "uuid-договора",
  "apartment": "uuid-квартиры",
  "client": "uuid-клиента",
  "cashbox": "uuid-кассы",
  "shift": "uuid-смены",
  "amount": "1000000.00",
  "comment": "Оплата по договору №123",
  "files": []  // опционально, см. раздел 4
}
```

Правила:
- заявка создаётся со статусом `pending` (ожидает одобрения кассы);
- сумма и реквизиты договора/клиента/квартиры сохраняются в заявке и используются при создании CashFlow;
- если продажа отменена до одобрения кассой — заявка должна быть помечена как `cancelled` (или аналог).

### 2.2 Обработка заявки кассой

Оператор кассы видит список заявок (см. раздел 3) и по заявке типа `apartment_sale` может:
- одобрить — создать приход в кассу;
- отклонить — отказать в проведении платежа.

При **одобрении**:
1. Проверить, что заявка в статусе `pending`.
2. Создать `CashFlow`:
   - `type = "income"`
   - `status = "approved"`
   - `amount` = сумма из заявки
   - `cashbox`, `shift` — из заявки (или переопределяются кассиром)
   - `source_business_operation_id` = ID договора/операции продажи
3. Обновить статус заявки: `approved`.
4. Обновить статус продажи/договора (если требуется, например «оплачено» или «частично оплачено»).
5. Сохранить прикреплённые файлы (см. раздел 4).

При **отклонении**:
1. Пометить заявку как `rejected`.
2. Никаких CashFlow не создавать.
3. Сохранить причину отклонения (например, поле `reject_reason`).

## 3. API заявок на кассу (общий, включая продажи и рассрочки)

Для всех типов заявок (продажа квартиры, первоначальный взнос по рассрочке, платеж по рассрочке и т.д.) используется единый API.

### 3.1 Список заявок

- **`GET /cash-register/requests/`**

Фильтры:
- `request_type` — `apartment_sale` | `installment_initial_payment` | `installment_payment` | ...
- `status` — `pending` | `approved` | `rejected` | `cancelled`
- `cashbox`, `shift`, `residential_complex`, `treaty`, `client` — по необходимости.

Ответ (пример полей):
- `id`
- `request_type`
- `status`
- `amount`
- `cashbox`, `shift`
- `source` (объект/идентификаторы: договор, квартира, клиент)
- `created_at`, `updated_at`
- `files` (список прикреплённых файлов, см. раздел 4)

### 3.2 Одобрить заявку

- **`POST /cash-register/requests/{id}/approve/`**

Тело (опционально):

```json
{
  "cashbox": "uuid-кассы",
  "shift": "uuid-смены",
  "paid_at": "2026-03-10T12:00:00+06:00",
  "comment": "Оплата получена"
}
```

Поведение:
1. Проверить, что заявка в статусе `pending`.
2. В зависимости от `request_type`:
   - `apartment_sale` — создать приход в кассу и отметить оплату по договору;
   - `installment_initial_payment` — создать приход и отметить, что первоначальный взнос по рассрочке оплачен;
   - `installment_payment` — создать приход и учесть оплату платежа по графику рассрочки.
3. Создать `CashFlow` (`type=income`, `status=approved`) с привязкой к бизнес-операции (`source_business_operation_id`).
4. Обновить статус заявки на `approved`.
5. Прикреплённые файлы заявки привязать к созданному CashFlow (или хранить на заявке, но отображать рядом с движением).

### 3.3 Отклонить заявку

- **`POST /cash-register/requests/{id}/reject/`**

Тело (опционально):

```json
{
  "reason": "Клиент отказался от оплаты"
}
```

Поведение:
1. Проверить, что заявка в статусе `pending`.
2. Обновить статус заявки на `rejected`.
3. Никаких CashFlow не создавать.

## 4. Файлы для заявок и движений по кассе

Для **любой заявки на кассу** и **любого движения по кассе (CashFlow)** должна быть возможность прикреплять файлы (чеки, акты, сканы и т.п.).

### 4.1 Файлы к заявке на кассу

- **Загрузка файла к заявке:**
  - `POST /cash-register/requests/{id}/files/` (multipart/form-data)

Поля:
- `file` (обязательно)
- `title` (опционально)

Ответ:
- список файлов заявки с полями:
  - `id`
  - `file_url`
  - `title`
  - `created_at`

Файлы, прикреплённые к заявке до одобрения, должны быть доступны кассиру при просмотре заявки.

### 4.2 Файлы к движению по кассе (CashFlow)

- **Загрузка файла к движению:**
  - `POST /cash-register/cashflows/{id}/files/` (multipart/form-data)

Поля:
- `file` (обязательно)
- `title` (опционально)

Ответ:
- список файлов, аналогично файлам заявки.

Логика:
- При одобрении заявки можно:
  - либо автоматически «привязать» файлы заявки к созданному CashFlow;
  - либо кассир может дополнительно загрузить новые файлы уже к самому CashFlow (например, скан чека после печати).

Рекомендуется в ответах на `GET /cash-register/requests/{id}/` и `GET /cash-register/cashflows/{id}/` возвращать список файлов, чтобы фронт мог их отобразить.

## 5. Рассрочка: первоначальный взнос и выплаты

### 5.1 Заявка на первоначальный взнос

При оформлении **договора в рассрочку**:
- если есть **первоначальный взнос**, вместо немедленного создания CashFlow создаётся **заявка типа `installment_initial_payment`** через тот же эндпоинт:

```json
{
  "request_type": "installment_initial_payment",
  "treaty": "uuid-договора",
  "apartment": "uuid-квартиры",
  "client": "uuid-клиента",
  "cashbox": "uuid-кассы",
  "shift": "uuid-смены",
  "amount": "300000.00",
  "comment": "Первоначальный взнос по договору №123",
  "files": []
}
```

Дальше заявка обрабатывается так же, как и обычная продажа: кассир одобряет/отклоняет, может прикрепить файлы; при одобрении создаётся приход в кассу и обновляется статус по рассрочке (первоначальный взнос оплачен).

### 5.2 Заявки на выплаты по рассрочке

Каждый платёж по графику рассрочки (ежемесячный, ежеквартальный и т.п.) также можно проводить через **заявку на кассу**:
- `request_type = "installment_payment"`.

Тело запроса (пример):

```json
{
  "request_type": "installment_payment",
  "treaty": "uuid-договора",
  "installment_schedule_item": "uuid-элемента-графика",
  "client": "uuid-клиента",
  "cashbox": "uuid-кассы",
  "shift": "uuid-смены",
  "amount": "50000.00",
  "comment": "Оплата по графику за апрель",
  "files": []
}
```

При одобрении:
1. Создать приход в кассу (CashFlow).
2. Пометить соответствующий элемент графика как оплаченный (полностью или частично, если поддерживаются частичные оплаты).
3. Сохранить/привязать файлы (чеки и т.п.) к движению.

## 6. Общие требования по логам и целостности

- Все действия по заявкам и движениям кассы (создание, одобрение, отклонение, загрузка файлов) должны логироваться:
  - кто (user_id), когда (timestamp), какое действие.
- При ошибках (недостаток средств, закрытая смена, неверная касса и т.п.) API должны возвращать стандартные ошибки (`400`/`403`/`409` и т.д.) с понятным сообщением.
- Связи между заявками, CashFlow и бизнес-объектами (договора, рассрочки, продажи) должны быть однозначными (по `source_business_operation_id` и дополнительным полям типа `treaty`, `installment_schedule_item`).

## 7. Закупки и склад — файлы

Для модулей закупок и склада нужно поддержать прикрепление файлов к основным сущностям.

### 7.1 Файлы к закупкам

- **Загрузка файлов к документу закупки:**
  - `POST /procurements/{id}/files/` (multipart/form-data)

Поля:
- `file` (обязательно)
- `title` (опционально)

Файлы могут быть: счета, договоры, накладные от поставщика и т.п.  
Ответ — список файлов с `id`, `file_url`, `title`, `created_at`.

### 7.2 Файлы к приёмке на склад

При **принятии закупки на склад** (документ приёмки/поступления на склад) также должна быть возможность прикреплять файлы:

- `POST /warehouse-receipts/{id}/files/` (или аналогичный урл по фактической модели, multipart/form-data)

Поля:
- `file` (обязательно)
- `title` (опционально)

Файлы: акты приёмки, сканы подписанных накладных и т.п.

## 8. Процесс работ, подрядчики, склад и касса

### 8.1 Процесс работ — дополнительные поля и файлы

В сущности **процесса работ** (BuildingWorkEntry / Task по объекту) добавить:
- `contractor` — ссылка на подрядчика (см. раздел 10);
- `contract_amount` — сумма договора с подрядчиком;
- `contract_term_start`, `contract_term_end` — сроки выполнения работ;
- `status` — статусы процесса работ (например: `planned`, `in_progress`, `paused`, `completed`, `cancelled`) для отслеживания хода работ;
- возможность прикреплять файлы:
  - `POST /work-entries/{id}/files/` (multipart/form-data, по аналогии с покупками/кассой).

Файлы: договор с подрядчиком, ТЗ, акты выполненных работ и т.п.

### 8.2 Запрос материалов со склада из процесса работ

Из записи процесса работ пользователь может создать **заявку на выдачу материалов со склада**:

- `POST /work-entries/{id}/warehouse-requests/`

Тело (пример):

```json
{
  "warehouse": "uuid-склада",
  "items": [
    {"nomenclature": "uuid-позиции", "quantity": "10.00", "unit": "шт"},
    {"nomenclature": "uuid-позиции-2", "quantity": "5.00", "unit": "м2"}
  ],
  "comment": "Материалы для отделочных работ"
}
```

Правила:
- заявка должна быть одобрена складом (ответственным по складу) перед фактической выдачей;
- склад может частично одобрить или отклонить позицию (по бизнес-логике модуля склада).

### 8.3 Выполнение / отмена работ и акт сверки материалов

При **выполнении** или **отмене** процесса работ нужно сформировать **акт сверки материалов**:

- `POST /work-entries/{id}/reconciliation-act/`

Тело (пример):

```json
{
  "returned_items": [
    {"nomenclature": "uuid-позиции", "quantity": "2.00", "unit": "шт"}
  ],
  "comment": "Возврат остатка материалов",
  "files": []
}
```

Поведение:
- фиксируется возврат остатков на склад (склад должен одобрить/провести этот акт по своим правилам);
- возможно хранить отдельную сущность «Акт сверки» со связью на процесс работ и операции склада;
- должна быть возможность прикреплять файлы к акту сверки:
  - `POST /work-entries/{id}/reconciliation-files/` или отдельный эндпоинт по ID акта.

### 8.4 Заявка на кассу для оплаты подрядчику

При переводе процесса работ в статус **`completed`** (и при наличии `contractor` + `contract_amount`) backend должен сформировать **заявку на кассу для оплаты подрядчику**:

- тип заявки, например: `contractor_payment`

Пример тела для создания заявки (внутренний вызов в `/cash-register/requests/`):

```json
{
  "request_type": "contractor_payment",
  "work_entry": "uuid-процесса-работ",
  "contractor": "uuid-подрядчика",
  "cashbox": "uuid-кассы",
  "shift": "uuid-смены",
  "amount": "150000.00",
  "comment": "Оплата по договору с подрядчиком",
  "files": []
}
```

Дальше заявка обрабатывается кассой по тому же механизму, что и для продаж/рассрочек: одобрение → расходный CashFlow, отклонение → только изменение статуса заявки.

Поля:

Файлы: договоры, лицензии, сертификаты, акты выполненных работ и т.п.

### 10.4 История процессов работ подрядчика

Нужно иметь возможность получать историю процессов работ, в которых участвовал подрядчик:

- `GET /contractors/{id}/work-history/`

Фильтры (опционально):
- `residential_complex`
- `status` процесса работ
- период по датам (создания, начала, завершения)

Ответ (пример):
- список записей процессов работ с полями:
  - `work_entry_id`
  - `title` / `description`
  - `residential_complex`
  - `contract_amount`
  - `status`
  - `term_start`, `term_end`
  - ключевые даты и, при необходимости, суммы оплат по кассе (агрегировано).

Это позволит видеть, какие работы подрядчик выполнял, по каким объектам и на какие суммы.

## 11. Поставщики (Suppliers) и закупки

Нужен отдельный модуль/эндпоинт для поставщиков строительных материалов, связанный с закупками и движением по кассе/складу.

### 11.1 Модель поставщика

Эндпоинты:
- список/создание: `GET /suppliers/`, `POST /suppliers/`
- детали/обновление/удаление: `GET /suppliers/{id}/`, `PATCH /suppliers/{id}/`, `DELETE /suppliers/{id}/`

Пример полей поставщика:

```json
{
  "company_name": "ОсОО СтройСнаб",
```

### 11.2 Файлы поставщика

- `POST /suppliers/{id}/files/` (multipart/form-data)

Поля:
- `file` (обязательно)
- `title` (опционально)

Файлы: договора с поставщиком, прайс-листы, сертификаты качества и т.п.

### 11.3 Настройка материалов поставщика

`supplied_materials` должно быть **совместимо с номенклатурой склада** и с позициями, которые мы добавляем вручную в закупках.

Варианты хранения:
- как массив в самом объекте поставщика (обновление через `PATCH /suppliers/{id}/`);
- или через отдельные эндпоинты:
  - `POST /suppliers/{id}/materials/`
  - `PATCH /supplier-materials/{id}/`
  - `DELETE /supplier-materials/{id}/`

Рекомендуемый состав полей по материалу:
- `name` — человекочитаемое название;
- `brand` — бренд/производитель;
- `unit` — единица измерения (должна совпадать с единицами в модуле склада);
- `price`, `currency` — цена и валюта;
- `warehouse_nomenclature` — ссылка на номенклатуру склада (uuid), если позиция уже заведена в справочнике склада;
- при необходимости дополнительные поля (артикул, код и т.п.).

При создании/редактировании **строк закупки** те же поля (номенклатура, unit, price, currency) используются и для:
- позиций, выбранных из `supplied_materials` поставщика;
- позиций, добавленных вручную **без поставщика** — в этом случае можно либо указать `warehouse_nomenclature`, либо создать новую номенклатуру по данным строки закупки.

### 11.4 Выбор поставщиков и товаров в закупке

В документе закупки (`/procurements/`) нужно поддержать:

- выбор **одного или нескольких поставщиков** в рамках одной закупки;
- выбор **товаров из списка поставщика** (по `suppliers.{id}.supplied_materials`) и возможность добавлять позиции **без поставщика** (например, разовая покупка или неформальный поставщик).

Возможные модели:
- в строке закупки (`procurement_line`) хранить:
  - `supplier` (опционально, uuid поставщика);
  - `warehouse_nomenclature` (ссылка на номенклатуру склада — общий справочник товаров);
  - количество, цену, валюту;
  - ссылку на `supplier_material` (если выбрано из справочника поставщика).

Таким образом:
- **товары поставщиков** и **товары, добавляемые вручную**, используют одну и ту же модель номенклатуры склада;
- в одном документе закупки можно иметь строки от разных поставщиков и строки без поставщика, но все они сопоставимы по складу.

Для самой закупки (`/procurements/`) дополнительно ввести поле **`contract_number`** (номер договора/заказа на закупку), а при передачах/возвратах со склада — хранить ссылку на исходный документ/номер договора для аудита.

### 11.5 Связь закупки, кассы, поставщика и файлов

При оплате закупки через кассу (заявки/движения по кассе):
- в `CashFlow` и/или заявке на кассу (`cash-register/requests`) должно быть поле `supplier`, если операция связана с поставщиком;
- все связанные **файлы** (документ закупки, файлы к заявке на кассу, файлы к CashFlow) позволяют восстановить историю взаимодействия с поставщиком;
- история действий по поставщику (заказы, оплаты, возвраты) может быть доступна через:
  - `GET /suppliers/{id}/purchase-history/` (по аналогии с `work-history` у подрядчиков), включая:
    - список закупок, сумм, оплат, статусов и прикреплённых файлов.

### 11.6 Движение по складу после приёмки

Все **принятые позиции** из документа приёмки (см. 7.2) должны формировать движения по складу:
- приход товара на конкретный склад и номенклатуру;
- сохранив связь с документом закупки, поставщиком и, при необходимости, кассовой операцией.

Дальнейшие операции по складу (браку, списанию, передаче) должны:
- создавать отдельные движения по складу;
- позволять прикреплять файлы к каждой операции;
- сохранять историю по каждой позиции и складу.

### 11.7 Брак, списание и передача товаров со склада

Из склада (Warehouse) должны быть доступны действия:

- **Брак / списание:**
  - `POST /warehouse-movements/write-off/`
  - Тело: склад, номенклатуры, количества, причина, файлы.
  - Результат: уменьшение остатка на складе, создание движения типа `write_off`, возможность прикрепить файлы (акты брака и т.п.).

- **Передача товаров подрядчику или в процесс работ:**
  - `POST /warehouse-movements/transfer-to-contractor/`
  - `POST /warehouse-movements/transfer-to-work-entry/`
  - Тело: склад, номенклатуры, количества, ссылка на `contractor` или `work_entry`, комментарий, файлы.
  - Результат: списание со склада и фиксация связи с подрядчиком или процессом работ (для последующей сверки и анализа затрат).

Для всех этих операций:
- должен вестись единый журнал движений склада;
- к каждому движению поддерживается прикрепление файлов (по аналогии с `cashflows/{id}/files/`):
  - `POST /warehouse-movements/{id}/files/` (multipart/form-data).
