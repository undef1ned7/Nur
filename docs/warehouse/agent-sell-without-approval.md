# Склад: право агента «Продажа без одобрения»

**Статус:** фронт готов, ждёт реализации на бэке.
**Страница:** `/crm/warehouse/agents` (сектор «Склад»).
**Дата:** 2026-07-09.

## 1. Задача

Сейчас поток продажи через агента выглядит так:

1. Агент собирает заявку на товар (`agent cart`, статус `draft`) и отправляет её — `POST /api/warehouse/agent-carts/{id}/submit/` → статус `submitted`.
2. Владелец/админ одобряет — `POST /api/warehouse/agent-carts/{id}/approve/` → статус `approved`, товар списывается со склада и зачисляется на остаток агента, после чего агент может продавать.

Нужно дать владельцу возможность отметить **конкретного доверенного агента**, чтобы его заявки **не требовали одобрения**: при отправке заявка сразу одобряется, товар сразу зачисляется агенту, и он может продавать без участия владельца.

## 2. Модель данных

В модель членства агента в компании (запись, которую возвращает
`GET /api/warehouse/agents/company-requests/` — там, где уже есть
`common_access_enabled`, `common_warehouse`, `can_sell_wholesale`,
`assigned_warehouse`) добавить поле:

```python
class CompanyWarehouseAgent(models.Model):
    ...
    can_sell_wholesale = models.BooleanField(default=False)          # уже есть
    can_sell_without_approval = models.BooleanField(default=False)   # НОВОЕ
```

- **Тип:** boolean, `default=False`, `null=False`.
- **Миграция:** обычная, все существующие агенты получают `False`
  (поведение не меняется, фича строго opt-in).
- Поле живёт на **членстве** (агент ↔ компания), а не на пользователе:
  один и тот же пользователь может быть доверенным агентом в одной
  компании и обычным — в другой.

## 3. Изменения в API

### 3.1. Сериализатор членства — вернуть поле

Во всех местах, где сериализуется `CompanyWarehouseAgent`, добавить
`can_sell_without_approval` в ответ:

- `GET /api/warehouse/agents/company-requests/` (список, вкл. фильтр `?status=active`)
- ответы `POST .../accept/`, `PATCH .../common-access/`,
  `POST /api/warehouse/agents/company-memberships/`

Пример элемента ответа:

```json
{
  "id": "3f0e6a2c-…",
  "user": "9b1caf00-…",
  "user_display": "Азамат Т.",
  "user_email": "agent@example.com",
  "status": "active",
  "assigned_warehouse": null,
  "common_access_enabled": true,
  "common_warehouse": "77aa10de-…",
  "can_sell_wholesale": true,
  "can_sell_without_approval": true,
  "created_at": "2026-07-01T10:00:00Z",
  "updated_at": "2026-07-09T12:30:00Z"
}
```

Фронт читает это поле в таблице активных агентов (колонка «Продажа без
одобрения») — если его нет в ответе, чекбокс всегда показывается выключенным.

### 3.2. `PATCH /api/warehouse/agents/company-requests/{id}/common-access/`

Endpoint уже принимает `common_access_enabled`, `common_warehouse`,
`can_sell_wholesale`. Добавить поддержку частичного обновления:

```json
{ "can_sell_without_approval": true }
```

Правила:

- Поле опциональное; если не передано — не менять.
- Доступ: только **owner/admin той же компании** (как для остальных полей
  этого endpoint'а). Агент сам себе флаг поставить не может.
- Менять можно только у членства со статусом `active` — для
  `pending/rejected/removed` возвращать `400` с понятным `detail`.
- Ответ — обновлённая запись членства (см. 3.1).

Фронт уже шлёт именно такой запрос при переключении чекбокса в таблице
(обработчик `handleCompanyNoApprovalChange` в
`src/Components/Sectors/Warehouse/Agents/Agents.jsx`).

### 3.3. `POST /api/warehouse/agents/company-memberships/`

Прямое назначение агента владельцем. Добавить опциональное поле в body:

```json
{
  "user": "9b1caf00-…",
  "assigned_warehouse": null,
  "common_access_enabled": true,
  "common_warehouse": "77aa10de-…",
  "can_sell_wholesale": false,
  "can_sell_without_approval": true
}
```

По умолчанию `false`. Фронт уже отправляет это поле из формы
«Назначить агентом» (чекбокс «Продажа без одобрения»).

### 3.4. `POST /api/warehouse/agent-carts/{id}/submit/` — ключевое изменение

Текущее поведение: `draft → submitted`, ждёт решения владельца.

Новое поведение:

```
submit:
  проверить, что cart.status == "draft" и вызывающий — автор заявки (агент)
  membership = активное членство агента в компании склада заявки
  если membership.can_sell_without_approval == False:
      как сейчас: status = "submitted"                     # ничего не меняется
  иначе (флаг включён):
      в ОДНОЙ транзакции выполнить ту же логику, что и approve:
          - проверить остатки по всем позициям заявки
          - списать товар со склада
          - зачислить товар на остаток агента (agent stock)
          - status = "approved", проставить approved_at
          - в поле "кем одобрено" записать признак автоодобрения
            (например, decided_by = None + auto_approved = True,
             или decided_by = system; главное — чтобы в истории было видно,
             что владелец вручную не одобрял)
      вернуть заявку со status = "approved"
```

Требования:

- **Атомарность:** проверка остатков + списание + смена статуса — одна
  транзакция с блокировкой строк остатков (как в существующем `approve`).
  Никаких промежуточных состояний «submitted, но товар списан».
- **Флаг читается в момент submit,** а не при создании заявки: если владелец
  выключил флаг, пока заявка была в `draft`, — заявка уйдёт в обычный
  `submitted`.
- **Формат ошибки нехватки остатков — тот же, что у `approve`:**

  ```json
  HTTP 400
  { "items": ["«Цемент М500»: на складе 3, запрошено 10", "…"] }
  ```

  Фронт уже показывает такие ошибки алертом «Недостаточно товара на складе»
  и для submit.
- Существующие endpoint'ы `approve/`, `reject/`, `dispatch/` не меняются:
  для заявки, автоодобренной на submit, повторный `approve` должен, как и
  сейчас, отдавать `400` («заявка не в статусе submitted»).

### 3.5. Что НЕ входит в объём

- Автоодобрение **возвратов** (`agent-return-carts`) — не трогаем, возвраты
  по-прежнему подтверждает владелец.
- Право на опт (`can_sell_wholesale`) — независимый флаг, логика не меняется.
- `POST /agent-carts/{id}/create-sale/` — без изменений: он и так работает
  только по одобренной заявке, автоодобренная заявка проходит по той же ветке.

## 4. Права и безопасность

| Действие | Кому разрешено |
|---|---|
| Читать `can_sell_without_approval` | owner/admin компании; агент — только своё членство |
| Менять флаг (`common-access`, `memberships`) | только owner/admin компании |
| Автоодобрение на submit | только автор заявки с активным членством и включённым флагом |

- Агент **не должен** иметь возможность передать
  `can_sell_without_approval` ни в каком доступном ему endpoint'е
  (например, при создании заявки в компанию) — сериализатор для агента
  должен игнорировать/запрещать это поле.
- При `remove` (отстранение агента) флаг можно не сбрасывать, но членство
  становится неактивным, поэтому автоодобрение перестаёт работать
  автоматически (проверка «активное членство» в submit обязательна).
- Если у агента задан `assigned_warehouse`, submit с автоодобрением должен
  работать только для заявок по этому складу (существующая проверка склада
  в submit/approve сохраняется).

## 5. Аудит

Записывать в существующий механизм истории/аудита заявок событие
автоодобрения, чтобы владелец мог отличить:

- `approved` вручную (кем, когда);
- `approved` автоматически по праву `can_sell_without_approval`
  (когда; «кем» — система/сам агент).

Минимум: `decided_at` заполняется, `decided_by` остаётся `NULL` (или
отдельное поле `auto_approved: true` в сериализаторе заявки).
