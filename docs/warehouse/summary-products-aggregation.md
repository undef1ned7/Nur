# Сводка (warehouse/summaries): потеря товаров в таблице «Товары»

Баг раздела «Сводка» (`/crm/warehouse/documents/sale?tab=summary`).
Обнаружен июль 2026 на реальной сводке **СВ-000019 «Тест»** (дата 2026-07-22).

## Симптом

Товар присутствует в накладной (в таблице «Накладные» сводки), но
**отсутствует в сводной таблице «Товары»**. Итоги (`totals.total_amount`,
`total_quantity`) при этом тоже расходятся с суммой по накладным.

## Причина (на бэкенде)

Снапшот сводки отдаёт две независимые структуры:

- `documents[]` — накладные с позициями `documents[].items[]` (полные, корректные);
- `products[]` — агрегат товаров для таблицы №1.

Фронт **не пересчитывает** `products[]` — он рендерит его как есть
(`normalizeSummary` вызывается без сырых накладных; см.
`src/Components/Sectors/Warehouse/Documents/components/Summary/summaryAggregation.js`).
Значит, позиции теряются именно при генерации `products[]` в снапшоте на бэкенде.

Наиболее вероятная причина — **агрегация по неполному ключу**: например,
группировка по `product_id` или только по названию. Тогда:

- позиции без `product_id` (свободный ввод, «сырые» позиции) выпадают или схлопываются;
- один и тот же товар с **разной ценой** в разных накладных схлопывается в одну
  строку (или, наоборот, теряется), а сумма считается неверно.

В данных СВ-000019 такой кейс есть: **«Май Карона Изобилия 5 л»** идёт по цене
`924.00` (SALE-...-0001) и по `860.00` (SALE-...-0004, -0005). Это прямой
кандидат на потерю/схлопывание при агрегации по названию/`product_id`.

## Требуемое поведение

`products[]` в снапшоте (и при `regenerate`) должен быть **точной агрегацией всех
позиций всех накладных** этой сводки. Инвариант:

> Множество (товар × ед. × цена) в `products[]` = множество тех же ключей,
> просуммированное по всем `documents[].items[]`. Ни одна позиция не теряется.

### Ключ агрегации

Группировать строку товара по кортежу **(наименование, единица, цена)** — так же,
как это делает эталонная клиентская агрегация
(`aggregateProducts` в `summaryAggregation.js`):

```
key = (name, unit, price)
```

- `name` — наименование позиции как в накладной (для свободного ввода — текст позиции,
  не только справочный товар);
- `unit` — единица измерения (нормализованная, напр. lower-case);
- `price` — цена за единицу. **Разные цены → разные строки** (не схлопывать).

Для каждой группы:

- `quantity` = сумма `quantity` позиций;
- `amount` = сумма `amount` позиций (а не `quantity * price` заново — учитывать скидки);
- `weight` = сумма `weight` позиций;
- `per_package` / `packages` — по данным товара (`packages = floor(quantity / per_package)`).

### Итоги

`totals` пересчитать от полного `products[]`:

- `products_count` = число строк в `products[]`;
- `total_quantity` = Σ `products[].quantity`;
- `total_weight` = Σ `products[].weight`;
- `total_amount` = Σ `products[].amount`.

Должно совпадать с итогами по `documents[]` (Σ `documents[].amount` и т.д.).

## Критерии приёмки

1. Для СВ-000019: каждая позиция из `documents[].items[]` присутствует в `products[]`.
2. «Май Карона Изобилия 5 л» представлена **двумя** строками — по 924.00 и по 860.00.
3. `totals.total_amount` = сумме `amount` по всем накладным (для СВ-000019 сверить
   с суммой всех `documents[].amount`).
4. Пересборка (`POST /warehouse/summaries/{id}/regenerate/`) даёт тот же полный
   `products[]`.
5. Позиции со свободным вводом (без `product_id`) не теряются.

## Проверка на фронте

Фронт корректен и правок не требует — он отображает `products[]`/`documents[]`
как пришли. Эталон правильной агрегации: `aggregateProducts()` в
`src/Components/Sectors/Warehouse/Documents/components/Summary/summaryAggregation.js`.
После фикса бэкенда таблицы «Товары» и «Накладные» в UI и PDF должны сходиться
без изменений клиента.
