# Склад: Зарплата агентов (процент с продаж по складам)

**Статус:** фронт готов, ждёт реализации на бэке.
**Страница:** `/crm/warehouse/salary` (сектор «Склад»), пункт меню «Зарплата».
**Дата:** 2026-07-10.
**Фронт:** `src/Components/Sectors/Warehouse/Salary/Salary.jsx`, API-слой `src/api/warehouseSalary.js`.

## 1. Задача

Владелец склада назначает каждому складу свою процентную ставку вознаграждения
агента — **отдельно для розничных и отдельно для оптовых продаж**. Когда агент
продаёт товар, который был получен с этого склада, ему автоматически
начисляется процент от суммы продажи по ставке склада-источника:

- продажа розничная (`is_wholesale = false`) → ставка `retail_percent`;
- продажа оптовая (`is_wholesale = true`) → ставка `wholesale_percent`.

Владелец видит историю начислений по всем агентам, сводку «кому сколько
должны» и фиксирует выплаты. Агент видит только свои начисления и выплаты.

## 2. Модель данных

Все модели принадлежат компании (мультитенантность как у остальных
warehouse-моделей). Все денежные поля — `Decimal(12, 2)`, проценты —
`Decimal(5, 2)` (0.00–100.00).

### 2.1. `WarehouseSalaryRate` — ставка склада

```python
class WarehouseSalaryRate(models.Model):
    warehouse = models.OneToOneField(Warehouse, on_delete=models.CASCADE,
                                     related_name="salary_rate")
    retail_percent = models.DecimalField(max_digits=5, decimal_places=2,
                                         default=Decimal("0.00"))
    wholesale_percent = models.DecimalField(max_digits=5, decimal_places=2,
                                            default=Decimal("0.00"))
    updated_at = models.DateTimeField(auto_now=True)
    updated_by = models.ForeignKey(User, null=True, on_delete=models.SET_NULL)
```

- Ставка `0.00` = процент не начисляется (запись может отсутствовать —
  трактовать как нули).
- Валидация: `0 <= percent <= 100`, иначе `400`.

### 2.2. `AgentSalaryAccrual` — начисление

```python
class AgentSalaryAccrual(models.Model):
    class Status(models.TextChoices):
        PENDING = "pending"     # продажа создана, но не оплачена (долг)
        ACCRUED = "accrued"     # продажа оплачена — к выплате
        PAID = "paid"           # закрыто выплатой
        CANCELED = "canceled"   # продажа отменена/возвращена

    class SaleType(models.TextChoices):
        RETAIL = "retail"
        WHOLESALE = "wholesale"

    company = models.ForeignKey(Company, on_delete=models.CASCADE)
    agent = models.ForeignKey(User, on_delete=models.CASCADE)          # продавец-агент
    sale = models.ForeignKey(AgentSale, on_delete=models.CASCADE,
                             related_name="salary_accruals")
    warehouse = models.ForeignKey(Warehouse, on_delete=models.PROTECT) # склад-источник
    sale_type = models.CharField(choices=SaleType.choices)
    sale_amount = models.DecimalField(max_digits=12, decimal_places=2) # база начисления
    percent = models.DecimalField(max_digits=5, decimal_places=2)     # СНИМОК ставки
    amount = models.DecimalField(max_digits=12, decimal_places=2)     # sale_amount * percent / 100
    status = models.CharField(choices=Status.choices)
    payout = models.ForeignKey("AgentSalaryPayout", null=True, blank=True,
                               on_delete=models.SET_NULL,
                               related_name="accruals")
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
```

Ключевые правила:

- **`percent` — снимок ставки в момент продажи.** Последующее изменение
  ставки склада НЕ пересчитывает существующие начисления.
- `amount = round(sale_amount * percent / 100, 2)`, округление
  `ROUND_HALF_UP`.
- Если ставка склада для данного типа продажи `0.00` — начисление
  **не создаётся вообще** (не плодим нулевые записи).
- На одну продажу может быть **несколько начислений**, если товар в продаже
  пришёл агенту с разных складов (см. 3.2) — по одному на каждый
  склад-источник. `sale_amount` в этом случае — часть суммы продажи,
  относящаяся к товарам этого склада.

### 2.3. `AgentSalaryPayout` — выплата

```python
class AgentSalaryPayout(models.Model):
    company = models.ForeignKey(Company, on_delete=models.CASCADE)
    agent = models.ForeignKey(User, on_delete=models.CASCADE)
    amount = models.DecimalField(max_digits=12, decimal_places=2)
    comment = models.CharField(max_length=255, blank=True, default="")
    created_by = models.ForeignKey(User, null=True, on_delete=models.SET_NULL,
                                   related_name="+")
    created_at = models.DateTimeField(auto_now_add=True)
```

## 3. Бизнес-логика начисления

### 3.1. Когда создаётся начисление

Начисление создаётся **в момент создания продажи агента** (существующий поток:
агент продаёт со своего остатка, продажи видны в
`GET /api/main/agents/me/sales/`; продажа из заявки —
`POST /api/warehouse/agent-carts/{id}/create-sale/`). Продажа несёт флаг
`is_wholesale` — он определяет тип (`retail`/`wholesale`) и какую из двух
ставок применять.

Статус начисления привязан к статусу продажи:

| Событие продажи | Статус начисления |
|---|---|
| Продажа создана, оплачена сразу (`paid`) | `accrued` |
| Продажа создана в долг (`debt` / не оплачена) | `pending` |
| Долг погашен (продажа стала `paid`) | `pending → accrued` |
| Возврат/отмена продажи (`canceled`) | `pending/accrued → canceled` |
| Возврат уже выплаченного начисления (`paid`) | статус оставить `paid`, создать **корректирующее начисление** с отрицательным `amount` и статусом `accrued` (уменьшит баланс агента) |

Частичный возврат: скорректировать пропорционально — уменьшить
`sale_amount`/`amount` начисления, если оно ещё `pending/accrued`; если уже
`paid` — отрицательное корректирующее начисление на сумму возврата × процент.

### 3.2. Как определяется склад-источник

Ставка берётся **со склада, с которого товар попал на остаток агента**
(момент одобрения заявки `agent-cart`: товар списывается со склада заявки и
зачисляется агенту). Для этого остаток агента должен хранить склад-источник:

- если в модели агентского остатка (agent stock) поля склада-источника ещё
  нет — добавить `source_warehouse = FK(Warehouse)` и заполнять его при
  approve заявки (склад заявки уже известен);
- при продаже каждая позиция трассируется на склад-источник своего остатка;
- позиции продажи группируются по складу-источнику → по одному начислению на
  склад: `sale_amount` = сумма позиций этого склада, `percent` = ставка этого
  склада для типа продажи.

Если по историческим остаткам склад-источник определить нельзя (старые
записи до миграции) — начисление не создаётся; в лог/аудит писать причину.

Если у агента `common_access_enabled` и продажи идут с «общего склада»
(`common_warehouse`) — источником считается этот склад, его ставка и
применяется.

### 3.3. Явно вне объёма

- Продажи владельца/сотрудников через документы (`warehouse/documents/sale`)
  — процент НЕ начисляется, только продажи агентов.
- Возвраты агента на склад (`agent-return-carts`) — не влияют на начисления
  (товар просто возвращается, продажи не было).
- Индивидуальные ставки «агент × склад» — не в этой итерации (ставка одна на
  склад для всех агентов). Модель это не блокирует: при необходимости позже
  добавится таблица переопределений.
- Интеграция с кассой (расходный ордер при выплате) — не в этой итерации,
  выплата фиксируется только в модуле зарплаты.

### 3.4. Баланс и выплаты

- `accrued_total` = Σ `amount` начислений в статусах `accrued` + `paid`
  за период (отрицательные корректировки включаются).
- `paid_total` = Σ `amount` выплат за период.
- `balance` («к выплате») = Σ `amount` всех начислений в статусе `accrued`
  (без фильтра по периоду — долг копится с начала).
- `POST /payouts/`: `amount` должен быть `> 0` и `<= balance` агента, иначе
  `400 {"amount": ["Сумма превышает баланс агента (1 234.00)"]}`.
- После создания выплаты начисления агента в статусе `accrued` закрываются
  **FIFO (от старых к новым)**: начисление помечается `paid` и получает
  `payout`, пока накопленная сумма закрытых начислений не покроет `amount`
  выплаты. Если выплата покрывает начисление частично — начисление
  **разбить**: закрытая часть (`paid`, привязана к выплате) + остаток
  (`accrued`). Всё в одной транзакции.

## 4. API

Базовый префикс: `/api/warehouse/salary/`. Все ответы списков — стандартная
DRF-пагинация `{count, next, previous, results}`. Все endpoints требуют
аутентификацию; принадлежность к компании проверяется как в остальных
warehouse-endpoint'ах.

### 4.1. `GET /api/warehouse/salary/rates/` — ставки складов

Доступ: **owner/admin** (агенту и сотруднику без права — `403`).

Возвращает **все склады компании**, включая те, у которых ставка ещё не
задана (для них нули) — фронт рисует таблицу настроек по этому списку.

Параметры: `search` (по названию склада, опционально), `page`.

```json
{
  "count": 2,
  "next": null,
  "previous": null,
  "results": [
    {
      "warehouse": "77aa10de-…",
      "warehouse_name": "Основной склад",
      "retail_percent": "5.00",
      "wholesale_percent": "2.50",
      "updated_at": "2026-07-10T09:00:00Z"
    },
    {
      "warehouse": "8c1f33ab-…",
      "warehouse_name": "Филиал Ош",
      "retail_percent": "0.00",
      "wholesale_percent": "0.00",
      "updated_at": null
    }
  ]
}
```

### 4.2. `PUT /api/warehouse/salary/rates/{warehouse_id}/` — изменить ставку

Доступ: **owner/admin**. Идемпотентный upsert: создаёт запись, если её нет.

```json
// request
{ "retail_percent": "5.00", "wholesale_percent": "2.50" }

// response 200 — элемент в формате 4.1
```

Ошибки: `400` при значении вне `[0, 100]` или не-числе
(`{"retail_percent": ["Процент должен быть от 0 до 100"]}`), `404` — чужой
или несуществующий склад.

### 4.3. `GET /api/warehouse/salary/accruals/` — история начислений

Доступ: **owner/admin** — все начисления компании; **агент** — только свои
(фильтр по `request.user` принудительно, параметр `agent` игнорируется);
сотрудник с правом `can_view_salary` — как admin, без права — `403`.

Параметры фильтрации (все опциональные):

| Параметр | Формат | Смысл |
|---|---|---|
| `date_from`, `date_to` | `YYYY-MM-DD` | период по `created_at` (включительно) |
| `agent` | uuid пользователя | только этот агент |
| `warehouse` | uuid склада | только этот склад-источник |
| `sale_type` | `retail` \| `wholesale` | тип продажи |
| `status` | `pending` \| `accrued` \| `paid` \| `canceled` | статус |
| `search` | строка | по номеру продажи |
| `page` | int | пагинация |

Сортировка по умолчанию: `-created_at`.

```json
{
  "count": 120,
  "next": "…?page=2",
  "previous": null,
  "results": [
    {
      "id": "a41b6c00-…",
      "created_at": "2026-07-10T14:21:00Z",
      "agent": "9b1caf00-…",
      "agent_name": "Азамат Т.",
      "sale": "5d2e9f10-…",
      "sale_number": "S-000123",
      "warehouse": "77aa10de-…",
      "warehouse_name": "Основной склад",
      "sale_type": "retail",
      "sale_amount": "12500.00",
      "percent": "5.00",
      "amount": "625.00",
      "status": "accrued"
    }
  ]
}
```

Фронт читает поля ровно с такими именами; `agent_name`, `sale_number`,
`warehouse_name` — денормализованные для отображения.

### 4.4. `GET /api/warehouse/salary/summary/` — сводка

Доступ: как в 4.3 (агент получает сводку только по себе, `by_agent`
содержит одну строку).

Параметры: `date_from`, `date_to`, `agent`, `warehouse` (как в 4.3).

```json
{
  "totals": {
    "accrued_total": "18400.00",   // начислено за период (accrued + paid)
    "paid_total": "12000.00",      // выплачено за период
    "balance": "6400.00",          // текущий долг перед агентами (весь, не за период)
    "accruals_count": 87           // число начислений за период
  },
  "by_agent": [
    {
      "agent": "9b1caf00-…",
      "agent_name": "Азамат Т.",
      "retail_amount": "10100.00",     // начислено по рознице за период
      "wholesale_amount": "2300.00",   // начислено по опту за период
      "accrued_total": "12400.00",
      "paid_total": "8000.00",
      "balance": "4400.00"             // текущий долг (не за период)
    }
  ]
}
```

### 4.5. `GET /api/warehouse/salary/payouts/` — история выплат

Доступ: owner/admin/сотрудник с правом — все; агент — только свои.

Параметры: `date_from`, `date_to`, `agent`, `page`. Сортировка `-created_at`.

```json
{
  "count": 3,
  "results": [
    {
      "id": "c1d2e3f4-…",
      "created_at": "2026-07-01T10:00:00Z",
      "agent": "9b1caf00-…",
      "agent_name": "Азамат Т.",
      "amount": "8000.00",
      "comment": "Аванс за июнь",
      "created_by": "0f9e8d7c-…",
      "created_by_name": "Нурбек У."
    }
  ]
}
```

### 4.6. `POST /api/warehouse/salary/payouts/` — выплатить агенту

Доступ: **только owner/admin** (сотруднику с `can_view_salary` — по решению
бэка; фронт кнопку показывает только owner/admin). Агент — `403`.

```json
// request
{
  "agent": "9b1caf00-…",
  "amount": "4400.00",
  "comment": "Зарплата за июнь"   // опционально
}

// response 201 — созданная выплата в формате 4.5
```

Логика — см. 3.4 (проверка баланса, FIFO-закрытие начислений, одна
транзакция). Ошибки:

- `400 {"amount": ["Сумма превышает баланс агента (4 400.00)"]}`;
- `400 {"agent": ["Агент не найден или не активен в компании"]}` — `agent`
  должен быть пользователем с активным членством агента в компании
  (`CompanyWarehouseAgent.status == "active"`) либо агентом, у которого есть
  начисления в компании.

## 5. Права и доступы

На фронте пункт меню «Зарплата» показывается при `profile.can_view_salary
=== true` (поле уже используется в других секторах — убедиться, что оно
отдаётся в профиле и для сектора «Склад»). В настройках доступов сотрудника
(секция «Склад») добавлен чекбокс «Зарплата» → `can_view_salary`.

| Действие | owner / admin | сотрудник с `can_view_salary` | агент |
|---|---|---|---|
| Видеть/менять ставки складов (4.1, 4.2) | ✅ | ❌ (`403`) | ❌ (`403`) |
| История начислений (4.3) | все | все | только свои |
| Сводка (4.4) | все | все | только своя |
| История выплат (4.5) | все | все | только свои |
| Создать выплату (4.6) | ✅ | ❌ | ❌ |

- **Агент видит свои начисления без права `can_view_salary`** — это его
  собственные деньги; бэк определяет агента по активному членству. Если
  профиль агента не содержит `can_view_salary=true`, пункт меню у него не
  появится — для агентов сектора «Склад» отдавать `can_view_salary: true`
  в профиле (или всегда `true` для роли agent).
- Агент не может видеть чужие начисления, ставки складов и создавать выплаты
  ни при каких параметрах запроса.

## 6. Целостность и краевые случаи

- **Транзакции:** создание продажи + начислений — одна транзакция; выплата +
  закрытие начислений — одна транзакция.
- **Идемпотентность:** повторная обработка одного события оплаты/возврата не
  должна дублировать начисления (уникальность по продаже+складу, кроме
  корректирующих записей).
- **Смена ставки** влияет только на продажи, созданные после изменения
  (снимок `percent` в начислении).
- **Удаление склада** — начисления сохраняются (`on_delete=PROTECT` или
  зарезервировать вариант с сохранением имени склада).
- **Отстранение агента** (`remove`) — история начислений и долг сохраняются;
  выплата отстранённому агенту допустима (долг надо закрыть).
- Продажа с нулевой суммой / полностью скидочная → начисление не создаётся.
- Часовой пояс фильтров дат — как в остальной аналитике склада
  (`date_from`/`date_to` трактуются в TZ компании).

## 7. Что уже сделано на фронте

- Страница `/crm/warehouse/salary` (`Salary.jsx`), три вкладки:
  - **Начисления** — фильтры (период, агент, склад, розница/опт, статус,
    поиск), карточки итогов (`totals`), таблица «Итоги по агентам»

  - **Выплаты** — история выплат, модалка «Выплатить агенту»
    (`POST payouts/`), подстановка баланса из `summary.by_agent`.
- Агенту вкладка ставок и кнопка выплаты не показываются, заголовок «Моя
  зарплата».
- Пункт меню «Зарплата» (`can_view_salary`) в `warehouseMenu.js`; чекбокс
  «Зарплата» в доступах сотрудника: `employeeAccessLabels.js`,
  `AccessList.jsx`, `Building/Employess/index.jsx` (секция «Склад»).
- Пока endpoints отвечают `404`, страница показывает баннер «Раздел ещё не
  активирован на сервере» — после деплоя бэка всё заработает без изменений
  фронта.
