# Консалтинг — Воронка продаж: аудит CRM-логики

**Страница:** `/crm/consulting/funnel` (фронт:
`src/Components/Sectors/Consulting/Funnel/Funnel.jsx`, thunks
`src/store/creators/funnelThunk.js`).
**Статус:** фронт зрелый (доски, drag&drop, WebSocket, роли). Ниже — что
исправлено на фронте и что бэкенд **обязан** гарантировать по бизнес-логике.

## 1. Исправлено на фронте

1. **Обновление доски после действий из карточки лида.** Раньше `Взять`
   (claim), `Вернуть в пул` (release), `Назначить` (assign), `Успех` (win)
   обновляли доску только через WebSocket. При недоступном WS карточка «зависала»
   в старом состоянии. Теперь эти действия дополнительно вызывают
   `onBoardRefresh()`.
2. **Убран мёртвый no-op** в `onDropToStage` (пустой `if` без тела) —
   поведение задокументировано комментарием: перенос на завершающую стадию
   разрешён и сотруднику, ограничение на повторное перемещение уже завершённого
   лида проверяется через `canDragLead()`.

## 2. Жизненный цикл лида (эталон)

```
[Общий пул]  --claim-->  [Мой]  --move-stage-->  … --> [Завершено/Won]
     ^                     |                                  |
     |                     +--release--> [Общий пул]          |
     +---------- assign (менеджер) ----------------------------+
[Мой] --lose--> [Проигран]
```

Статусы: `new → in_progress → won | lost`. Стадии (`funnel-stages`) настраиваемые;
системные стадии (`isSystemStage`) защищены от удаления/переименования.

## 3. Что бэкенд обязан гарантировать

### 3.1. Права и владение

- `claim`: лид без владельца (`owner=null`) может взять любой сотрудник с
  доступом к воронке. Гонки: два claim одновременно — второй получает `409`.
- `release`: только владелец или менеджер; возвращает `owner=null`.
- `assign`: только менеджер/owner; ставит `owner=<user>`.
- Перемещение по стадиям (`move-stage`) — владелец или менеджер; проверять
  `allowedTransitions` (нельзя прыгать через запрещённые переходы, если они
  заданы).
- Завершённый лид (won/lost или стадия «Завершено») повторно двигать может
  только менеджер (фронт уже ограничивает через `canDragLead`).

### 3.2. Сайд-эффекты при завершении лида (критично)

При переходе лида в **выигрыш** (`win` или перенос на завершающую стадию) сервер
атомарно (одна транзакция):

1. Создаёт/находит **клиента** (если у лида есть `client` — использует его).
2. Создаёт **сделку** (`deal`) на сумму по услуге/тарифу **с учётом роли**
   (см. [services-role-pricing.md](./services-role-pricing.md)). Абонентский
   тариф → создаёт расписание абонентских платежей (для абонентской матрицы,
   [subscription-matrix.md](./subscription-matrix.md)).
3. Начисляет **зарплату** продавцу по ставке услуги
   ([salary-auto-accrual.md](./salary-auto-accrual.md)).
4. Пишет запись в **аналитику** (продажа появляется в `/consalting/sales/`).
5. Рассылает реалтайм-события: обновление доски + персональное уведомление
   владельцу ([realtime-notifications.md](./realtime-notifications.md)).

Идемпотентность: повторный `win` того же лида не должен дублировать сделку/
начисление (проверка по `lead_id`).

### 3.3. Оплата по лиду

`POST /consalting/leads/{id}/register-payment/` (`payment_mode` ∈
`cash|transfer|debt|installment`) — создаёт сделку и, для `debt/installment`,
расписание платежей. Требует наличие клиента у лида (иначе `400` с понятным
`detail`). Фронтовый fallback уже реализован в `funnelThunk.registerLeadPayment`.
