# Консалтинг — Реалтайм: WebSocket + пуш (персонально пользователю)

**Фронт:** общий per-user сокет `useNotificationsSocket`
(`/ws/notifications/?token=<accessToken>`), смонтирован в `Header.jsx` и работает
во всех секторах. Для консалтинга добавлен хук
`src/Components/Sectors/Consulting/common/useConsultingRealtime.js`, который
реагирует на «мои» события и обновляет данные + показывает десктоп-пуш.
**Статус:** транспорт уже есть. Нужно, чтобы бэкенд **слал персональные события**
по бизнес-логике (см. ниже).

## 1. Ключевое требование — адресность

Уведомление и обновление приходят **тому пользователю, которого касается
событие**, а не всем. Пример: лид назначен сотруднику А (авто-распределение или
менеджер) → **только А** получает уведомление и обновление списка; сотрудник Б
ничего не получает.

Сокет `/ws/notifications/` авторизуется JWT, т.е. бэкенд точно знает `user` и
`company` соединения. Отправлять событие нужно **в персональную группу
пользователя** (напр. Channels-группа `user_<user_id>`), а не в общую группу
компании.

## 2. Когда слать событие (бизнес-логика)

| Событие | Кому | `type` |
|---|---|---|
| Лид из WhatsApp авто-назначен сотруднику | назначенному сотруднику | `consulting.lead.assigned` |
| Менеджер вручную назначил лид | назначенному сотруднику | `consulting.lead.assigned` |
| Лид из воронки назначен/передан мне | новому владельцу | `consulting.funnel.lead.assigned` |
| Мне поручена задача по лиду | исполнителю | `consulting.lead.task.assigned` |
| Моя сделка закрыта / начислена зарплата | продавцу | `consulting.salary.accrued` |

Не слать эти события всей компании. Менеджеру/owner отдельные копии — по желанию
(например, сводное «назначен лид сотруднику А»), но это **другой** получатель.

## 3. Формат сообщения

Сокет уже понимает такой конверт (см. `useNotificationsSocket.handleMessage`):

```jsonc
{
  "type": "consulting.lead.assigned",
  "data": {
    "id": "notif-uuid",           // обязателен (дедуп)
    "title": "Вам назначен лид",
    "message": "Иван, +996700… — из WhatsApp",
    "type": "consulting.lead.assigned",  // дублируем внутри — фронт матчит по нему
    "is_read": false,
    "created_at": "2026-07-23T10:12:00",
    "meta": { "lead_id": "…", "funnel_id": "…" }   // необязательно
  }
}
```

- Поле верхнего уровня `type` или `data.type` фронт использует, чтобы понять, что
  это событие лида/воронки, и обновить соответствующий экран
  (`useConsultingRealtime` матчит по вхождению `lead|лид|funnel|assign`).
- `id` обязателен — по нему идёт дедуп в слайсе `notification` и защита от
  повторного срабатывания.
- Уведомление также попадает в колокольчик в шапке и проигрывает звук — это уже
  реализовано, отдельных действий не требуется.

## 4. Что делает фронт при получении

- **Страница «Лиды»** (`Leads.jsx`): при событии с `lead|лид` — перезагружает
  список входящих (для текущего пользователя) и показывает десктоп-пуш, если
  вкладка не в фокусе.
- **Воронка** (`Funnel.jsx`): при событии с `lead|лид|funnel|assign` —
  обновляет доски и показывает десктоп-пуш.
- Десктоп-пуш (Web Notifications API) — best-effort: разрешение запрашивается при
  открытии экрана; показывается только когда вкладка скрыта, чтобы не дублировать
  видимый интерфейс.

## 5. Опционально — счётчик непрочитанных

Если бэкенд шлёт `{"type":"unread_count","count":N}` — фронт обновит бейдж
колокольчика (уже поддержано). Полезно слать после каждого персонального события.

## 6. Чек-лист

- [ ] Событие уходит в группу конкретного пользователя, не в группу компании.
- [ ] У каждого сообщения уникальный `id`.
- [ ] `type` содержит `lead`/`funnel`, чтобы экран консалтинга обновился.
- [ ] Сотрудник Б не получает уведомлений о лидах сотрудника А.
