# Building (Строительство) — Backend-спецификация (кратко)

Базовый URL: `/api/building/`

Документ описывает требования для «сферы строительство»:
- **блоки квартир** (как этажи) + фильтрация;
- **типы договоров** для строительства + фильтрация;
- **группы/папки для договоров** (иерархия, перенос, фильтры);
- **рассрочка**: перенос переплаты на следующий период с фиксацией в выплатах.

---

## 1) Квартиры — блоки (Blocks) + фильтрация

### 1.1 Модель

Отдельную сущность **Block** не делаем. Блок задаётся напрямую в квартире как обычное текстовое поле (ввод вручную).

В `Apartment` добавить поле:
- `block` (string, nullable) — например: `Блок А`, `Подъезд 1`, `Секция 2`.

Рекомендации по значению:
- хранить как строку (с сохранением регистра);
- при фильтрации нормализовать (trim) и/или хранить ещё `block_normalized` (lowercase) — по реализации.

### 1.2 Фильтрация квартир по блокам

Эндпоинт квартир:
- `GET /apartments/?residential_complex=<uuid>&block=<string>`

Дополнительно можно поддержать:
- `block__in=<string,string>` (мультивыбор) — если используется DRF filterset.

### 1.3 Агрегация «как этажи»

Нужно реализовать агрегатор по блокам **аналогично этажам**:

- `GET /objects/{residential_complex_id}/blocks/stats/`

Ответ (пример):

```json
[
  {"block": "Блок А", "total": 40, "available": 30, "reserved": 6, "sold": 4},
  {"block": "Блок Б", "total": 32, "available": 20, "reserved": 5, "sold": 7}
]
```

Статусы квартир — такие же, как сейчас (`available`, `reserved`, `sold`).

## 2) Договора — типы для строительства + фильтрация

### 2.1 Тип договора (treaty_type)

В договор (`BuildingTreaty`) добавить отдельное поле **`treaty_type`** (не путать с `operation_type`).

Перечисления:
- `construction_department` — строительный отдел
- `sale` — продажа
- `booking` — бронь
- `procurement` — закупки
- `other` — прочее

Рекомендации:
- `operation_type` оставить для «операции по квартире» (если уже используется: sale/booking/other).
- `treaty_type` использовать для бизнес-классификации договора в строительстве.

### 2.2 Создание/редактирование договора

- `POST /treaties/`
- `PATCH /treaties/{id}/`

Добавить/принимать поле:
- `treaty_type` (обязательно или с дефолтом — на ваше усмотрение)

### 2.3 Фильтрация по типам

В списке договоров:
- `GET /treaties/?treaty_type=<value>`
- `GET /treaties/?treaty_type__in=construction_department,sale` (если поддерживается)

Существующие фильтры (`residential_complex`, `client`, `status`, `payment_type`, `operation_type`, `apartment`, `search`) сохраняются.

## 3) Группы/папки договоров (иерархия)

### 3.1 Сущность TreatyGroup (папка)

Нужно добавить иерархические группы (папки) для договоров.

Модель:
- `id` (uuid)
- `residential_complex` (uuid, nullable) — если группы привязаны к ЖК; либо `null` для общих по компании
- `parent` (uuid, nullable) — родительская группа (для вложенности)
- `title` (string) — название папки
- `order` (int, опционально) — сортировка
- `is_active` (bool)

Правило вложенности:
- допускается вложенность N уровней;
- запрет циклов (`parent` не может быть потомком текущего).

### 3.2 API групп

- `GET /treaty-groups/` — список (в т.ч. деревом)
  - query: `residential_complex=<uuid>` (опционально)
  - query: `tree=true` (опционально) — вернуть вложенную структуру
- `POST /treaty-groups/` — создать
- `PATCH /treaty-groups/{id}/` — обновить (переименование, смена родителя, сортировка)
- `DELETE /treaty-groups/{id}/` — удалить (по правилам: либо запрет при наличии договоров/детей, либо каскад/перенос)

Пример ответа `tree=true`:

```json
[
  {
    "id": "uuid",
    "title": "Продажи",
    "children": [
      {"id": "uuid", "title": "2026", "children": []}
    ]
  }
]
```

### 3.3 Привязка договора к группе

В `BuildingTreaty` добавить поле:
- `group` (uuid, nullable) — папка договора.

### 3.4 Перемещение договоров между группами

Варианты:

**A) Простое обновление договора**
- `PATCH /treaties/{id}/` с `{"group": "uuid-группы"}` или `{"group": null}`.

**B) Массовый перенос**
- `POST /treaties/move/`

Тело:

```json
{
  "treaty_ids": ["uuid-1", "uuid-2"],
  "target_group": "uuid-группы-назначения"
}
```

### 3.5 Фильтрация договоров по группам

- `GET /treaties/?group=<uuid>`
- `GET /treaties/?group__in=<uuid,uuid>` (опционально)

Опционально: режим «включая подгруппы»:
- `GET /treaties/?group=<uuid>&include_descendants=true`

## 4) Рассрочка — перенос переплаты на следующий период

### 4.1 Текущее API платежей

Платежи по взносу рассрочки:
- `POST /treaty-installments/{installment_id}/payments/` с полем `amount`

### 4.2 Новое бизнес-правило (переплата)

Если пользователь оплачивает сумму больше, чем остаток по текущему взносу:

- `amount_to_apply_current = min(amount, current_remaining)`
- `overpay = amount - amount_to_apply_current`

**Поведение:**
1. В текущем взносе создать платеж на `amount_to_apply_current` (как и раньше), довести `paid_amount` до `amount` и перевести `installment.status` в `paid`.
2. Если есть `overpay > 0`:
   - автоматически применить переплату к следующему(им) взносу(ам) в порядке `order`/`due_date` (в рамках того же договора):
     - уменьшать переплату, пока не закончится или пока не кончатся взносы.
3. Для каждого затронутого следующего взноса:
   - создать **отдельную запись платежа** (чтобы в выплатах было видно, куда ушла сумма);
   - увеличить его `paid_amount`;
   - если взнос закрыт полностью — пометить `paid`.

Итог: один внесённый платеж может породить **несколько записей** в истории платежей (по нескольким периодам/взносам).

### 4.3 Ответ API

Рекомендуется вернуть агрегированный результат:

```json
{
  "applied": [
    {"installment_id": "uuid-1", "amount": "10000.00", "paid_total_after": "10000.00", "status": "paid"},
    {"installment_id": "uuid-2", "amount": "5000.00", "paid_total_after": "5000.00", "status": "planned"}
  ],
  "input_amount": "15000.00",
  "unused_amount": "0.00"
}
```

Если переплата больше суммы всех оставшихся взносов — вернуть `unused_amount > 0` и/или ошибку по бизнес-правилам (на ваш выбор). Чаще удобнее разрешить и вернуть `unused_amount`, чтобы фронт мог предложить сделать возврат/зачёт.

### 4.4 Идемпотентность (опционально)

Чтобы касса/клиент не создали дубль при повторной отправке:
- поддержать заголовок `Idempotency-Key` или поле `external_payment_id`;
- при повторе возвращать уже созданное распределение.
