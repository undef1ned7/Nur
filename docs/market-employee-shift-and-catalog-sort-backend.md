# Магазин: автозакрытие смены при удалении сотрудника + сортировка витрины

Два требования по сфере **Магазин (Market)**. Фронт по второму пункту уже готов.

---

## 1. Удаление сотрудника с открытой сменой → смена закрывается автоматически

**Проблема:** если в сфере Магазин удалить сотрудника (кассира), у которого есть **открытая
кассовая смена**, смена «зависает» открытой — касса/отчёты остаются в некорректном состоянии.

**Требование:** при удалении сотрудника бэкенд должен в той же транзакции **автоматически
закрыть** все его открытые смены.

### Поведение
- При `DELETE` сотрудника (или его деактивации/увольнении) найти все его смены со статусом
  «открыта» в рамках компании.
- Каждую такую смену **закрыть** так же, как обычное закрытие смены:
  - проставить время закрытия (`closed_at` = now), статус → «закрыта»;
  - зафиксировать итоговые суммы (наличные/безнал/итог по сменам), как при ручном закрытии;
  - закрытие должно отражаться в отчётах по сменам и кассе.
- Указать причину/пометку автозакрытия, напр. `close_reason = "employee_deleted"` или
  системный комментарий «Смена закрыта автоматически: сотрудник удалён».
- Выполнять в **одной транзакции** с удалением: если закрытие смены упало — удаление откатывается.
- Идемпотентность: если открытых смен нет — обычное удаление без изменений.

### Граничные случаи
- Несколько открытых смен у сотрудника — закрыть все.
- Если бизнес-правила запрещают удалять кассира с открытой сменой — альтернатива: вернуть `409`
  с понятным сообщением. **Предпочтительно** всё же автозакрытие (как просили).
- Логировать автозакрытие (кто инициировал удаление, какие смены закрыты).

### Чек-лист
- [ ] При удалении сотрудника открытые смены закрываются автоматически.
- [ ] Закрытие идентично штатному (итоги, время, статус, отчёты).
- [ ] Транзакционность: ошибка закрытия → откат удаления.
- [ ] Пометка причины автозакрытия + лог.

## 2. Сортировка витрины `/catalog/{slug}` (storefront)

Эндпоинт витрины: `GET /api/main/public/companies/{slug}/showcase/`.

Фронт уже отправляет параметр **`ordering`** и показывает выпадающий список сортировки
(Название А–Я / Я–А, Цена ↑/↓, Скидка ↑/↓). Нужно, чтобы бэкенд сортировал **по всей выборке**
(до пагинации), а не только в пределах страницы.

### Параметр `ordering`
| Значение | Сортировка |
|----------|-----------|
| `name` | Название А–Я |
| `-name` | Название Я–А |
| `final_price` | Цена по возрастанию (дешевле сначала) |
| `-final_price` | Цена по убыванию (дороже сначала) |
| `discount_percent` | Скидка по возрастанию |
| `-discount_percent` | Скидка по убыванию |

- `final_price` — цена с учётом скидки (та, что показывается на витрине). Если на витрине
  отображается именно она — сортировать по ней.
- Параметр опционален; без него — поведение по умолчанию (как сейчас).
- Сортировка применяется **вместе** с уже существующими `search`, `category`, `page`,
  `page_size` и **до** пагинации (иначе сортируется только текущая страница).
- Неизвестные значения `ordering` игнорировать (вернуть дефолтный порядок), не падать `400`.

### Чек-лист
- [ ] `showcase/` принимает `ordering` со значениями выше.
- [ ] Сортировка по всей выборке до пагинации.
- [ ] `final_price` и `discount_percent` доступны для сортировки.
- [ ] Совместимо с `search` / `category` / пагинацией.
- [ ] Неизвестный `ordering` → дефолт, без ошибки.

> Примечание: до готовности бэкенда фронт сортирует загруженную страницу локально (fallback),
> поэтому визуально сортировка работает сразу; полноценная сортировка по всем товарам появится
> после поддержки `ordering` на бэкенде.
