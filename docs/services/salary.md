# Услуги: Зарплата мастеров (процент с оказанных услуг)

**Статус:** фронт готов, ждёт реализации на бэке.
**Страница:** `/crm/services/salary` (сектор «Услуги», ключ сектора `services`),
пункт меню «Зарплата». По этой же логике и тем же компонентом уже сегодня можно
включить раздел для секторов «Барбершоп» (`/crm/barber/salary`) и «Стоматология»
(`/crm/dentistry/salary`) — маршруты для них зарегистрированы, но пункт меню
показан только в секторе «Услуги» (см. §7). Компонент один на все три сектора,
т.к. они используют один и тот же backend-домен `barbershop`.
**Дата:** 2026-07-22.
**Фронт:** `src/Components/Sectors/Barber/Salary/Salary.jsx`, API-слой
`src/api/servicesSalary.js`. Сделано по образцу `docs/warehouse/salary.md`
(«Склад: Зарплата агентов») — модели, статусы и порядок выплат идентичны,
изменена только логика определения ставки и источника начисления (см. ниже).

## 1. Задача

Владелец назначает **каждой услуге** свою процентную ставку вознаграждения —
единая ставка, без деления на розницу/опт (в отличие от склада). Когда
мастер завершает запись (`appointment`) клиента, ему автоматически
начисляется процент от стоимости оказанной услуги по ставке этой услуги.

Владелец видит историю начислений по всем мастерам, сводку «кому сколько
должны» и фиксирует выплаты. Мастер видит только свои начисления и выплаты.

## 2. Модель данных

Все модели принадлежат компании (мультитенантность как у остальных
barbershop-моделей: `Appointment`, `Service`, `Client`). Денежные поля —
`Decimal(12, 2)`, проценты — `Decimal(5, 2)` (0.00–100.00).

### 2.1. `ServiceSalaryRate` — ставка услуги

```python
class ServiceSalaryRate(models.Model):
    company = models.ForeignKey(Company, on_delete=models.CASCADE)
    service = models.OneToOneField(Service, on_delete=models.CASCADE,
                                   related_name="salary_rate")
    percent = models.DecimalField(max_digits=5, decimal_places=2,
                                  default=Decimal("0.00"))
    updated_at = models.DateTimeField(auto_now=True)
    updated_by = models.ForeignKey(User, null=True, on_delete=models.SET_NULL)
```

- Ставка `0.00` = процент не начисляется (запись может отсутствовать —
  трактовать как ноль).
- Валидация: `0 <= percent <= 100`, иначе `400`.
- В отличие от склада здесь **одна** ставка на услугу (нет розницы/опта) —
  у записи на услугу нет аналога «тип продажи».

### 2.2. `MasterSalaryAccrual` — начисление

```python
class MasterSalaryAccrual(models.Model):
    class Status(models.TextChoices):
        PENDING = "pending"     # зарезервировано, см. примечание ниже
        ACCRUED = "accrued"     # запись завершена — к выплате
        PAID = "paid"           # закрыто выплатой
        CANCELED = "canceled"   # запись отменена/статус завершения снят

    company = models.ForeignKey(Company, on_delete=models.CASCADE)
    master = models.ForeignKey(User, on_delete=models.CASCADE)              # мастер = appointment.barber
    appointment = models.ForeignKey(Appointment, on_delete=models.CASCADE,
                                    related_name="salary_accruals")
    service = models.ForeignKey(Service, on_delete=models.PROTECT)          # услуга-источник
    service_amount = models.DecimalField(max_digits=12, decimal_places=2)   # база начисления (см. 3.2)
    percent = models.DecimalField(max_digits=5, decimal_places=2)          # СНИМОК ставки
    amount = models.DecimalField(max_digits=12, decimal_places=2)          # service_amount * percent / 100
    status = models.CharField(choices=Status.choices)
    payout = models.ForeignKey("MasterSalaryPayout", null=True, blank=True,
                               on_delete=models.SET_NULL,
                               related_name="accruals")
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=["appointment", "service"],
                condition=~models.Q(status="canceled"),
                name="uniq_active_accrual_per_appointment_service",
            )
        ]
```

Ключевые правила:

- **`percent` — снимок ставки в момент завершения записи.** Последующее
  изменение ставки услуги НЕ пересчитывает существующие начисления.
- `amount = round(service_amount * percent / 100, 2)`, округление
  `ROUND_HALF_UP`.
- Если ставка услуги `0.00` — начисление **не создаётся вообще** (не плодим
  нулевые записи).
- На одну запись (`appointment`) может быть **несколько начислений** — по
  одному на каждую услугу из `appointment.services`, если их несколько (см.
  3.2, правило распределения суммы).
- **Про статус `pending`:** в отличие от склада (где продажа может уйти «в
  долг» до оплаты), у записи на услугу в текущей модели нет отдельного
  статуса оплаты — запись либо ещё не завершена (начисления нет вообще),
  либо `completed` (начисление сразу `accrued`). Статус `pending` в enum
  оставлен только для совместимости контракта между секторами и на случай,
  если позже появится отдельный флаг оплаты записи; **в первой итерации бэк
  его не использует** — начисление создаётся сразу в статусе `accrued`.

### 2.3. `MasterSalaryPayout` — выплата

```python
class MasterSalaryPayout(models.Model):

```

Полностью аналогично складу (см. `docs/warehouse/salary.md` §2.3, §3.4).

## 3. Бизнес-логика начисления

### 3.1. Когда создаётся начисление

Начисление создаётся **в момент, когда запись (`Appointment`) переходит в
статус `completed`** (существующий поток: смена статуса записи —
`PATCH /api/barbershop/appointments/{id}/` с полем `status`, см.
`src/Components/Sectors/Barber/Recorda/`). Мастер начисления — это
`appointment.barber` на момент завершения.

| Событие записи | Действие с начислением |
|---|---|
| Статус записи → `completed` (впервые) | Создать начисление(я) сразу в статусе `accrued` — по одному на каждую услугу из `appointment.services` с ненулевой ставкой (3.2) |
| Статус `completed` снят (админ откатил статус на `booked`/`confirmed` по ошибке) | Если начисления ещё `accrued` — перевести в `canceled`; если уже `paid` — оставить `paid`, создать **корректирующее начисление** с отрицательным `amount` и статусом `accrued` (уменьшит баланс мастера) |
| Статус записи → `canceled` / `no_show` (после того как была `completed`) | Аналогично строке выше — отмена/корректировка |
| Повторное завершение той же записи (`completed` → откат → снова `completed`) | Создать начисления заново (уникальность по `appointment+service` среди активных не даёт задублировать, см. `Meta.constraints` в 2.2) |
| Изменение `services` или `price` записи **после** того как начисления уже созданы (`accrued`/`paid`) | Существующие начисления **не пересчитываются** (снимок). Если по факту нужно исправить — вручную отменить старые и провести повторное завершение |

### 3.2. Как определяется `service_amount` (база начисления по каждой услуге)

У записи один общий `price` (может быть переопределён вручную и не обязан
равняться сумме базовых цен услуг — см. поле `price` в
`RecordaModal.jsx`, учитывает `discount`), но услуг в `appointment.services`
может быть несколько. Правило распределения:

- **Одна услуга в записи** → `service_amount = appointment.price` целиком.
- **Несколько услуг** → `appointment.price` распределяется пропорционально
  базовым ценам услуг на момент завершения записи:

  ```
  base_i = service_i.price  (текущая цена услуги в справочнике)
  service_amount_i = round(appointment.price * base_i / Σ base_j, 2)
  ```

  Если `Σ base_j == 0` (у всех услуг в записи цена не задана/нулевая) —
  разделить `appointment.price` поровну между услугами.
  Округление остатка (сумма частей может на копейку отличаться от
  `appointment.price` из-за округления) — отнести на услугу с наибольшей
  долей.
- Если `appointment.price` не задан (`null`) или `<= 0` — начисления не
  создаются ни по одной услуге записи.

### 3.3. Явно вне объёма

- Ставка привязана к услуге, а не к мастеру — индивидуальные ставки
  «мастер × услуга» не в этой итерации (модель это не блокирует, при
  необходимости позже добавится таблица переопределений, как и на складе).
- Интеграция с кассой (расходный ордер при выплате) — не в этой итерации,
  выплата фиксируется только в модуле зарплаты.
- Услуги, оказанные без записи в календаре (если такой сценарий появится) —
  вне объёма, начисление привязано к `Appointment`.

### 3.4. Баланс и выплаты

Полностью аналогично складу (`docs/warehouse/salary.md` §3.4):

- `accrued_total` = Σ `amount` начислений в статусах `accrued` + `paid` за
  период (отрицательные корректировки включаются).
- `paid_total` = Σ `amount` выплат за период.
- `balance` («к выплате») = Σ `amount` всех начислений в статусе `accrued`
  (без фильтра по периоду — долг копится с начала).
- `POST /payouts/`: `amount` должен быть `> 0` и `<= balance` мастера, иначе
  `400 {"amount": ["Сумма превышает баланс мастера (1 234.00)"]}`.
- После создания выплаты начисления мастера в статусе `accrued` закрываются
  **FIFO** (от старых к новым): начисление помечается `paid` и получает
  `payout`, пока накопленная сумма не покроет `amount` выплаты. Частичное
  покрытие → начисление **разбить**: закрытая часть (`paid`) + остаток
  (`accrued`). Всё в одной транзакции.

## 4. API

Базовый префикс: `/api/barbershop/salary/`. Все ответы списков — стандартная
DRF-пагинация `{count, next, previous, results}`. Все endpoints требуют
аутентификацию; принадлежность к компании проверяется как в остальных
`barbershop`-endpoint'ах.

### 4.1. `GET /api/barbershop/salary/rates/` — ставки услуг

Доступ: **owner/admin** (мастеру и сотруднику без права — `403`).

Возвращает **все услуги компании** (`is_active` и неактивные — фронт сам не
фильтрует), включая те, у которых ставка ещё не задана (для них нули).

Параметры: `search` (по названию услуги, опционально), `page`.

```json
{
  "count": 2,
  "next": null,
  "previous": null,
  "results": [
    {
      "service": "77aa10de-…",
      "service_name": "Стрижка мужская",
      "price": "800.00",
      "percent": "40.00",
      "updated_at": "2026-07-20T09:00:00Z"
    },
    {
      "service": "8c1f33ab-…",
      "service_name": "Окрашивание",
      "price": "2500.00",
      "percent": "0.00",
      "updated_at": null
    }
  ]
}
```

### 4.2. `PUT /api/barbershop/salary/rates/{service_id}/` — изменить ставку

Доступ: **owner/admin**. Идемпотентный upsert: создаёт запись, если её нет.

```json
// request
{ "percent": "40.00" }

// response 200 — элемент в формате 4.1
```

Ошибки: `400` при значении вне `[0, 100]` или не-числе
(`{"percent": ["Процент должен быть от 0 до 100"]}`), `404` — чужая или
несуществующая услуга.

### 4.3. `GET /api/barbershop/salary/accruals/` — история начислений

Доступ: **owner/admin** — все начисления компании; **мастер** — только свои
(фильтр по `request.user` принудительно, параметр `master` игнорируется);
сотрудник с правом `can_view_salary` — как admin, без права — `403`.

Параметры фильтрации (все опциональные):

| Параметр | Формат | Смысл |
|---|---|---|
| `date_from`, `date_to` | `YYYY-MM-DD` | период по `created_at` (включительно) |
| `master` | uuid пользователя | только этот мастер |
| `service` | uuid услуги | только эта услуга |
| `status` | `pending` \| `accrued` \| `paid` \| `canceled` | статус |
| `search` | строка | по номеру записи (`appointment_number`) |
| `page` | int | пагинация |

Сортировка по умолчанию: `-created_at`.

```json
{
  "count": 120,
  "next": "…?page=2",
  "previous": null,
  "results": [
    {
      "id": "a41b6c00-…",
      "created_at": "2026-07-20T14:21:00Z",
      "master": "9b1caf00-…",
      "master_name": "Азамат Т.",
      "appointment": "5d2e9f10-…",
      "appointment_number": "A-000123",
      "service": "77aa10de-…",
      "service_name": "Стрижка мужская",
      "service_amount": "800.00",
      "percent": "40.00",
      "amount": "320.00",
      "status": "accrued"
    }
  ]
}
```

Фронт читает поля ровно с такими именами; `master_name`,
`appointment_number`, `service_name` — денормализованные для отображения.
`appointment_number` — желателен для читаемости (аналог `sale_number` на
складе), но не обязателен: фронт при его отсутствии показывает
`appointment` (сырой id).

### 4.4. `GET /api/barbershop/salary/summary/` — сводка

Доступ: как в 4.3 (мастер получает сводку только по себе, `by_master`

```json
    {

      "accruals_count": 42,

      "accrued_total": "12400.00",
      "paid_total": "8000.00",
      "balance": "4400.00"
    }
  ]
}
```

### 4.5. `GET /api/barbershop/salary/payouts/` — история выплат

Доступ: owner/admin/сотрудник с правом — все; мастер — только свои.

Параметры: `date_from`, `date_to`, `master`, `page`. Сортировка
`-created_at`.

```json
{
  "count": 3,
  "results": [
    {
      "id": "c1d2e3f4-…",
      "created_at": "2026-07-01T10:00:00Z",
      "master": "9b1caf00-…",
      "master_name": "Азамат Т.",
      "amount": "8000.00",
      "comment": "Аванс за июнь",
      "created_by": "0f9e8d7c-…",
      "created_by_name": "Нурбек У."
    }
  ]
}
```

### 4.6. `POST /api/barbershop/salary/payouts/` — выплатить мастеру

Доступ: **только owner/admin** (фронт кнопку показывает только им). Мастер —
`403`.

```json
// request
{
  "master": "9b1caf00-…",
  "amount": "4400.00",
  "comment": "Зарплата за июнь"   // опционально
}

// response 201 — созданная выплата в формате 4.5
```

Логика — см. 3.4 (проверка баланса, FIFO-закрытие начислений, одна
транзакция). Ошибки:

- `400 {"amount": ["Сумма превышает баланс мастера (4 400.00)"]}`;
- `400 {"master": ["Мастер не найден или неактивен в компании"]}` — `master`
  должен быть пользователем с активным членством в компании (обычный
  сотрудник, `/users/employees/`), а не обязательно тем, кто когда-либо был
  назначен на запись.

## 5. Права и доступы

На фронте пункт меню «Зарплата» показывается при `profile.can_view_salary
=== true`. В отличие от секции «Склад» (где агент — отдельная сущность и
получает `can_view_salary: true` автоматически), в секторе «Услуги» мастер —
обычный сотрудник компании: право выдаётся **явно**, через чекбокс
«Зарплата» в настройках доступов сотрудника (секция «Барбершоп» — общая для
секторов «Услуги», «Барбершоп», «Стоматология», см. §7).

| Действие | owner / admin | сотрудник с `can_view_salary` | мастер без `can_view_salary` |
|---|---|---|---|
| Видеть/менять ставки услуг (4.1, 4.2) | ✅ | ❌ (`403`) | ❌ (`403`) |
| История начислений (4.3) | все | все | ❌ (пункт меню скрыт) |
| Сводка (4.4) | все | все | ❌ |
| История выплат (4.5) | все | все | ❌ |
| Создать выплату (4.6) | ✅ | ❌ | ❌ |

- Мастер без явно выданного `can_view_salary` **не видит свою зарплату** в
  этой версии (иначе, чем на складе для агентов) — так решено, т.к. в
  секторе «Услуги» нет отдельной роли «агент»: владелец сам решает, кому из
  сотрудников открыть раздел (например, только администраторам, а мастерам
  зарплату называют лично). Если продукту потребуется, чтобы мастер видел
  свою зарплату без права — обсудить отдельно, это меняет контракт профиля.
- Мастер (если ему всё же открыт `can_view_salary`) не может видеть чужие
  начисления, ставки услуг и создавать выплаты ни при каких параметрах
  запроса — фильтрация по `request.user` принудительная на бэке.

## 6. Целостность и краевые случаи

- **Транзакции:** завершение записи + создание начислений — одна
  транзакция; выплата + закрытие начислений — одна транзакция.
- **Идемпотентность:** повторная обработка одного события завершения не
  должна дублировать начисления (`UniqueConstraint` на `appointment+service`
  среди активных, см. 2.2).
- **Смена ставки** влияет только на записи, завершённые после изменения
  (снимок `percent` в начислении).
- **Удаление услуги** — начисления сохраняются (`on_delete=PROTECT` или
  зарезервировать вариант с сохранением имени услуги).
- **Увольнение/отстранение мастера** — история начислений и долг
  сохраняются; выплата уволенному мастеру допустима (долг надо закрыть).
- Запись с нулевой/неуказанной стоимостью → начисление не создаётся (3.2).
- Часовой пояс фильтров дат — как в остальной аналитике сектора «Услуги»
  (`date_from`/`date_to` трактуются в TZ компании).
- Секторы «Барбершоп» и «Стоматология» используют тот же API и модели
  (общий домен `barbershop`) — начисления не должны пересекаться между
  компаниями разных секторов, но в остальном логика идентична; отдельных
  различий по сектору в бэке заводить не нужно.

## 7. Что уже сделано на фронте

- Страница `/crm/services/salary` (`Salary.jsx`, общий компонент также
