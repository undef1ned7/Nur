# Зарплата сотрудников (Building Payroll) — Backend-спецификация

Базовый префикс: `/api/building/` (или актуальный префикс salary-модуля).

Права:
- начисления/управление: `can_view_building_salary` (или `owner/admin/superuser`)
- выплата через кассу (в т.ч. аванс): дополнительно `can_view_building_cash_register` (или `owner/admin/superuser`)

---

## 1. Общая логика

- Зарплата привязана к **ЖК (residential_complex)**. Период начислений создаётся **по конкретному ЖК**.
- При **создании периода** backend **автоматически** создаёт строки начислений по всем сотрудникам, **назначенным на этот ЖК** (members), с подстановкой оклада/ставки из настроек компенсации. Отдельный запрос на добавление строк по каждому сотруднику не нужен.
- Корректировки (премия, удержание, аванс) сохраняются по одной операции; после каждой операции пересчитываются и сохраняются итоги по строке (суммы сразу суммируются).
- **Аванс** при создании отправляет запрос в кассу (выплата). Если касса отказывает — аванс не считается проведённым, сумма к выплате по строке не уменьшается (операция откатывается или помечается как неуспешная).

## 2. Сотрудники и настройки оклада/ставки

- **Список сотрудников (для выбора/настроек):**  
  `GET /salary/employees/`  
  Опционально: фильтр по ЖК, чтобы показывать только сотрудников, назначенных на выбранный ЖК (например `?residential_complex=<uuid>`).

- Ответ по сотруднику: `id`, `display`, `compensation_id` (если есть), `salary_type`, `base_salary`, `is_active`, а также поля начисления от продаж (см. ниже): `sale_commission_type`, `sale_commission_value`.

- **Настройка оклада/ставки:**  
  `PATCH /salary/employees/{user_id}/compensation/`  

  Поля:
  - `salary_type` (`monthly` | `daily` | `hourly`), `base_salary`, `is_active`, `notes`.
  - **Начисление от продаж** (опционально):
    - `sale_commission_type`: `fixed` | `percent` | `none` (или `null`/пусто — начисление от продаж выключено).
    - `sale_commission_value`: число — при `fixed` сумма в валюте (например `5000.00`), при `percent` — доля от суммы продажи в процентах (например `2.5` для 2,5%). При `none` поле игнорируется.

Пример тела запроса с начислением от продаж:

```json
{
  "salary_type": "monthly",
  "base_salary": "45000.00",
  "is_active": true,
  "sale_commission_type": "percent",
  "sale_commission_value": "2.5"
}
```

или фикс с продажи:

```json
{
  "sale_commission_type": "fixed",
  "sale_commission_value": "3000.00"
}
```

Источник данных для автозаполнения строк периода: сотрудники, назначенные на ЖК (`/objects/{residential_complex_id}/members/`), и их компенсация (оклад/ставка). Если у сотрудника нет настроек компенсации, в строку подставляется `base_amount = 0` (или явно помечать в ответе).

### 2.1 Автоначисление от продаж

Если у сотрудника в компенсации заданы `sale_commission_type` и `sale_commission_value` (и тип не `none`):

- При **продаже** (создание/подписание договора, сделка по ЖК и т.п.), где этот сотрудник указан как ответственный (или менеджер сделки), backend **автоматически** создаёт корректировку типа **премия (`bonus`)** в строке начисления этого сотрудника.
- Строка начисления берётся из **текущего периода по этому ЖК**: период в статусе `draft` или `approved`, в который попадает дата продажи (например, период, у которого `period_start <= date_sale <= period_end`). Если подходящего периода нет — либо не создавать начисление, либо создавать при появлении периода (на усмотрение реализации).
- Сумма корректировки:
  - при `sale_commission_type: "fixed"`: `sale_commission_value` (фиксированная сумма за сделку);
  - при `sale_commission_type: "percent"`: `(сумма продажи) * (sale_commission_value / 100)` — сумма продажи берётся из договора/сделки (например, полная сумма договора или оплаченная часть — уточнить в бизнес-правилах).
- В корректировке сохранять связь с источником (например `source_treaty_id` или `source_sale_id`), чтобы было видно, за какую сделку начислено. В `comment` можно подставлять текст вида «От продажи № …» или «Процент от продажи …».
- **Любую такую корректировку** (в т.ч. автоматическую от продаж) **можно отменить** по общим правилам отмены корректировок (см. п. 5.3).

## 3. Периоды начислений (Payroll)

### 3.1 Создание периода

- `POST /salary/payrolls/`

Тело запроса (обязательно указать ЖК):

```json
{
  "title": "ЗП за март 2026",
  "period_start": "2026-03-01",
  "period_end": "2026-03-31",
  "residential_complex": "uuid-жк"
}
```

**Поведение backend при создании:**

1. Создать запись периода со статусом `draft`.
2. Получить список сотрудников, назначенных на данный ЖК (members с `is_active=true`).
3. Для каждого такого сотрудника создать **строку начисления** (payroll line):
   - `employee` = ID пользователя-сотрудника;
   - `base_amount` = значение из настроек компенсации сотрудника (`base_salary`), если настроек нет — `0.00`;
   - при необходимости: `comment` пустой или по умолчанию.
4. Итоги по строкам (см. ниже) пересчитать: пока корректировок нет — `net_to_pay = base_amount`, `paid_total = 0`, остаток к выплате = `base_amount`.

В ответе на `GET /salary/payrolls/{id}/` или при последующем запросе `GET /salary/payrolls/{id}/lines/` фронт получает период уже со списком строк (все сотрудники ЖК с окладами).

### 3.2 Список и CRUD периода

- Список: `GET /salary/payrolls/` (фильтр по `residential_complex` рекомендуется).
- Детали: `GET /salary/payrolls/{id}/`.
- Обновление: `PATCH /salary/payrolls/{id}/` — только в статусе `draft` (например, правка `title`, дат).
- Удаление: `DELETE /salary/payrolls/{id}/` — только в статусе `draft`. При удалении периода удаляются все связанные строки, корректировки и записи о выплатах (или каскад по БД).

Статусы периода:
- `draft` — редактирование разрешено, строки и корректировки можно менять.
- `approved` — начислено, выплаты и авансы разрешены (запросы в кассу разрешены).
- `paid` — автоматически, когда по всем строкам выплачено полностью (`paid_total >= net_to_pay`).

Начисление (перевод в «утверждён»):
- `POST /salary/payrolls/{id}/approve/` → период переводится в `status=approved`.

## 4. Строки начислений (Payroll Lines)

- Строки создаются **только автоматически** при создании периода (по списку сотрудников ЖК). Отдельный `POST` на создание строки по сотруднику опционален (если в продукте нужно вручную добавлять/удалять сотрудника в периоде).
- Список строк периода: `GET /salary/payrolls/{payroll_id}/lines/`.
- Детали строки: `GET /salary/payroll-lines/{id}/` (при необходимости).

По каждой строке хранятся и пересчитываются:
- `base_amount` — оклад/база из настроек на момент создания строки (или ручное значение, если поддерживается правка).
- Суммы по корректировкам: премии (+), удержания (-), авансы (-).
- **Итог к выплате по строке:** `net_to_pay = base_amount + сумма премий - сумма удержаний - сумма проведённых авансов` (формула уточняется в реализации; авансы, по которым касса отказала, в эту формулу не входят).
- `paid_total` — сумма уже проведённых выплат (из кассы).
- Остаток к выплате: `net_to_pay - paid_total`; не может стать отрицательным (валидация при выплатах и при авансах).

## 5. Корректировки (премия, удержание, аванс)

Каждая корректировка — отдельная операция: создаётся запись, после чего по строке пересчитываются суммы и сохраняются в строке (или в связанной агрегирующей таблице). То есть каждая операция сохраняется и сразу суммируется.

### 5.1 Добавление корректировки

- `POST /salary/payroll-lines/{line_id}/adjustments/`

Тело (пример):

```json
{
  "adjustment_type": "bonus",
  "amount": "5000.00",
  "comment": "Премия за март"
}
```

Типы:
- `bonus` — премия (увеличивает сумму к выплате).
- `deduction` — удержание (уменьшает сумму к выплате).
- `advance` — аванс (уменьшает сумму к выплате и требует проведения через кассу, см. ниже).

Поведение backend для `bonus` и `deduction`:
- Сохранить запись корректировки.
- Пересчитать по строке: суммы по типам, `net_to_pay`, остаток к выплате.
- Проверить, что остаток к выплате и итоговая логика не дают переплаты (например, `paid_total <= net_to_pay`). При нарушении — вернуть `400` с сообщением.

### 5.2 Аванс и интеграция с кассой

При создании корректировки с `adjustment_type: "advance"` backend должен:

1. Проверить, что период в статусе `approved` (авансы допустимы только после начисления).
2. Проверить, что сумма аванса не превышает остаток к выплате по строке: `amount <= (net_to_pay - paid_total)` (с учётом уже запланированных, но ещё не проведённых авансов — если такая модель есть).
3. Сохранить запись корректировки во временном/черновом состоянии (например, `status=pending` или не учитывать в `net_to_pay` до успешной выплаты — в зависимости от реализации).
4. Отправить запрос на выплату в кассу (тот же механизм, что и для обычной выплаты ЗП):
   - сумма = `amount` аванса;
   - касса, смена и т.д. — из тела запроса или правил (например, обязательные поля `cashbox`, опционально `shift`).

Если касса **подтверждает** выплату (например, создаётся CashFlow/операция расхода):
- Пометить корректировку аванса как проведённую (например, `status=completed`).
- Учесть аванс в пересчёте строки: уменьшить остаток к выплате, увеличить `paid_total` на сумму аванса (или связать аванс с записью выплаты).
- Вернуть клиенту успешный ответ.

Если касса **отказывает** (ошибка 4xx/5xx, недостаточно средств, смена закрыта и т.д.):
- **Откатить** аванс: не учитывать его в сумме к выплате. Варианты реализации:
  - удалить только что созданную запись корректировки аванса и вернуть клиенту `400`/`409` с текстом ошибки кассы; или
  - сохранить корректировку со статусом `rejected`/`failed` и не включать её в расчёт `net_to_pay` и `paid_total`, а в ответе вернуть ошибку.
- В любом случае: сумма к выплате по строке должна остаться такой, как до попытки аванса (аванс обратно «начисляется» к доступной сумме выплаты в смысле того, что остаток не уменьшается).

Клиенту в случае отказа кассы вернуть понятную ошибку (например, «Касса отклонила операцию: смена закрыта» / «Недостаточно средств в кассе»).

### 5.3 Отмена (удаление) корректировки

- `DELETE /salary/payroll-adjustments/{id}/`

**Должна быть возможность отменить любую корректировку** — ручную (премия, удержание) или автоматическую (в т.ч. начисление от продаж). Ограничения по статусу периода:

- Период в статусе **`draft`**: можно удалять любые корректировки (премии, удержания, непроведённые авансы).
- Период в статусе **`approved`**: можно отменять премии и удержания; по авансам — только те, по которым касса ещё не провела выплату (или отмена через отдельный сценарий возврата в кассе — на усмотрение реализации). Отмена корректировки «от продаж» допустима так же, как отмена обычной премии.

После отмены пересчитать по строке суммы и `net_to_pay`, `paid_total`. Если корректировка была привязана к сделке (`source_treaty_id` и т.п.), запись удаляется или помечается отменённой; повторное начисление по той же сделке не создавать (идемпотентность по источнику).

## 6. Выплаты (частичные) через кассу

- Список выплат по строке: `GET /salary/payroll-lines/{id}/payments/`.
- Создать выплату: `POST /salary/payroll-lines/{id}/payments/`  
  Тело: `amount`, `cashbox` (uuid), опционально `shift`, `paid_at`.

Правила:
- Выплаты разрешены только для периода в статусе `approved`.
- Сумма выплаты не должна превышать остаток: `amount <= (net_to_pay - paid_total)`.
- При успешной выплате создаётся запись в кассе (CashFlow, тип `expense`, статус `approved`, привязка к операции через `source_business_operation_id` = UUID выплаты), `paid_total` по строке увеличивается на `amount`.

Если запрос к кассе завершается ошибкой — выплата не создаётся (или создаётся со статусом «отклонено» и не увеличивает `paid_total`), клиенту возвращается ошибка.

## 7. Мои начисления (для сотрудника)

- `GET /salary/my/lines/` — текущий пользователь видит только свои строки начислений и выплат (по всем периодам/ЖК, к которым имеет доступ).

---

## 8. Итоговая схема по строке (для backend)

- **Хранить по строке:**  
  `base_amount`, агрегаты по корректировкам (или считать их по связям), `net_to_pay`, `paid_total`, при необходимости `balance_to_pay = net_to_pay - paid_total`.
- **После каждой операции** (добавление/удаление корректировки, проведение аванса, выплата): пересчёт и сохранение `net_to_pay`, `paid_total` и проверка правил (не переплата, не отрицательный остаток).
- **Аванс:** одна операция = одна запись корректировки + один запрос в кассу; при отказе кассы — аванс не уменьшает сумму к выплате (откат или статус «не проведён»).
