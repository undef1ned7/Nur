# Консалтинг: бэкенд-доработки (DnD, услуги по ролям, оплата абонентки)

Дополнение к `FUNNEL_FRONTEND.md` / `FUNNEL_ROLES_FRONTEND.md`.
Контракт по трём доработкам: сохранение порядка drag-and-drop, услуги по ролям,
оплата абонентской платы с карточки клиента.

Базовый префикс: `/api/consalting/` (раздел 3 частично `/api/main/`).
Авторизация: `Authorization: Bearer <token>`.

---

## 1. Сохранение порядка (drag-and-drop)

### 1.1. Порядок воронок per-user

Персональный порядок воронок-строк на странице. Хранится на сервере, у каждого
сотрудника свой.

```
GET /api/consalting/user-preferences/
```
**200:**
```json
{ "funnel_order": ["uuid1", "uuid2", "uuid3"] }
```
- Если предпочтений нет → `{ "funnel_order": [] }` (не ошибка).
- На отдаче сервер **отфильтровывает** удалённые и недоступные воронки,
  **сохраняя порядок**.

```
PATCH /api/consalting/user-preferences/
Body: { "funnel_order": ["uuid2", "uuid1", "uuid3"] }
```
**200:** `{ "funnel_order": ["uuid2", "uuid1", "uuid3"] }`
- Сервер дедуплицирует список (с сохранением порядка) и эхом возвращает сохранённое.
- `PUT` поддерживается как алиас `PATCH`.
- Работает только со своими предпочтениями (по токену).

> Воронки, которых нет в списке, фронт добавляет в конец сам — сервер хранит ровно
> то, что прислали.

### 1.2. Bulk-переупорядочивание стадий

Одним запросом вместо N×PATCH.

```
POST /api/consalting/funnel-stages/reorder/
Body: [
  { "id": "uuid-стадии", "order": 0 },
  { "id": "uuid-стадии", "order": 1 },
  { "id": "uuid-стадии", "order": 2 }
]
```
**200:**
```json
{ "updated": 3, "stages": [ /* обновлённые стадии, отсортированы по order */ ] }
```

Поведение и ошибки:

| Код | Когда |
|-----|-------|
| `200` | Успех, порядок применён в транзакции |
| `400` | В списке есть **системная** стадия (`is_system=true`) → `{ "detail": "...", "ids": [...] }` |
| `400` | Тело не список / пустое; конфликт порядка |
| `404` | Стадии не найдены или из другой компании → `{ "detail": "...", "ids": [...] }` |
| `403` | Нет `manage_stages` на воронке |

- Права — **те же, что у `PATCH /funnel-stages/{id}/`** (`manage_stages(funnel)`),
  проверяются для каждой затронутой воронки.
- Можно слать стадии **нескольких воронок** в одном запросе.
- Fallback на N×PATCH по-прежнему рабочий.

## 2. Услуги, привязанные к роли

У услуги появилось поле `custom_role` (FK на кастомную роль, nullable).

```
GET   /api/consalting/services/
POST  /api/consalting/services/
PATCH /api/consalting/services/{id}/
```

Объект услуги теперь содержит:
```json
{
  "id": "uuid",
  "name": "Консультация",
  "price": "5000.00",
  "installation_price": "0.00",
  "tariffs": [],
  "custom_role": "uuid-роли"   // или null = общая услуга
}
```

- `custom_role: null` → услуга **общая**, видна во всех воронках.
- `custom_role: <id>` → услуга относится к воронке этой роли.
- POST/PATCH принимают `custom_role` (можно `null`).
- Роль из **чужой компании** → `400 { "custom_role": "Роль принадлежит другой компании." }`.
- При **удалении роли** услуга не удаляется — `custom_role` обнуляется (становится общей).
- Бонус: серверный фильтр `GET /api/consalting/services/?custom_role=<uuid>` работает.

> Клиентская фильтрация на воронке роли остаётся как у вас:
> `(!service.custom_role || service.custom_role === funnel.custom_role)`.

## 3. Оплата абонентской платы с карточки клиента

### 3.1. Расписание абонентки теперь оплачиваемо

```
GET /api/main/clients/{id}/subscription-schedule/
```
**200** (обёртка `{ "items": [...] }` — читайте `.items`):
```json
{
  "items": [
    {
      "period": "2026-06",
      "period_label": "Июн 2026",
      "amount": "5000.00",
      "status": "planned",          // "planned" | "paid"
      "paid": false,
      "active": true,
      "deal": "uuid-сделки",         // ← теперь есть
      "installment_id": "uuid"       // ← конкретный период (взнос)
    }
  ]
}
```

Что изменилось:
- Каждый период подкреплён реальной сделкой (`ClientDeal kind=DEBT`) и взносом
  (`DealInstallment`), поэтому несёт `deal` и `installment_id`.
- `paid` / `status` отражают **фактическую** оплату взноса.
- **Скользящее окно:** сервер держит ~12 неоплаченных периодов вперёд и доращивает
  их со временем.
- Создание сделки **ленивое** — при первом GET расписания (в т.ч. для старых
  абонентских продаж).

> ⚠️ Обёртка ответа осталась `{ "items": [...] }`, а не голый массив (как было в
> исходной доке) — чтобы не ломать текущий контракт. Если нужен голый массив — скажите.

### 3.2. Проведение оплаты

Существующий эндпоинт сделок:
```
POST /api/main/clients/{clientId}/deals/{dealId}/pay/
Body: {
  "installment_id": "uuid",        // из плитки расписания
  "amount": "5000.00",             // опционально; по умолчанию остаток взноса
  "idempotency_key": "uuid",       // обязателен
  "note": ""
}
```
**200:** обновлённый объект сделки с графиком.

- `installment_id` закрывает именно этот период (частичная оплата поддержана).
- `idempotency_key` уважается — повтор с тем же ключом не плодит платёж.
- После оплаты период в расписании становится `paid: true` / `status: "paid"`
  (перезапросите `subscription-schedule`).

> `payment_mode` (`cash`/`transfer`/`card`) бэком сейчас **игнорируется** — оплата
> проходит, но способ не сохраняется. Скажите, если нужно записывать.

## Краткий чеклист интеграции для фронта

- [ ] Порядок воронок: при загрузке `GET /user-preferences/`, при drop — `PATCH`.
      Fallback на localStorage при `404`.
- [ ] Стадии: при drop — `POST /funnel-stages/reorder/`. Fallback на N×PATCH при `404`.
- [ ] Услуги: читать/слать `custom_role`; фильтрация на воронке роли заработает
      автоматически.
- [ ] Абонентка: брать `deal` + `installment_id` из `items`, неоплаченные плитки —
      кликабельные; оплата через `/deals/{id}/pay/`.
