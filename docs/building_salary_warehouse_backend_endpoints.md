# Building — Backend API (зарплата + склад, кратко)

## 1. Выплаты по зарплате через кассу

### 1.1 Создание выплаты по строке зарплаты

**Эндпоинты:**
- список выплат по строке: `GET /salary/payroll-lines/{id}/payments/`
- создать выплату: `POST /salary/payroll-lines/{id}/payments/`

**Тело `POST /salary/payroll-lines/{id}/payments/`:**

```json
{
  "amount": "20000.00",
  "cashbox": "uuid-кассы",
  "shift": "uuid-смены",
  "paid_at": "2026-03-10T12:00:00+06:00",
  "status": "approved"
}
```

**Поля:**
- `amount` (обязательно) — сумма выплаты.
- `cashbox` (uuid, обязательно) — касса.
- `shift` (uuid, опционально) — смена.
- `paid_at` (опционально) — дата/время выплаты.
- `status` (опционально): `draft` | `approved`. По умолчанию можно использовать `draft` или сразу `approved`.

**Логика при `status = "approved"`:**
- Проверка:
  - период по строке зарплаты в статусе `approved`;
  - `amount <= (net_to_pay - paid_total)` по данной строке.
- В одной транзакции:
  - увеличить `paid_total` строки на `amount`;
  - уменьшить остаток к выплате (`net_to_pay - paid_total`);
  - создать запись `CashFlow` (касса): тип `expense`, статус `approved`, `source_business_operation_id` = UUID выплаты.

Итого: указанный `amount` **снимается из суммы к выплате сотруднику** и добавляется в «выплачено» (`paid_total`), а касса получает связанное движение расхода.

**При `status = "draft"`:**
- запись создаётся без списания средств и без изменения `paid_total`;
- дальнейшее одобрение возможно отдельным действием (например, `PATCH /salary/payments/{id}/` или `POST /salary/payments/{id}/approve/`) — реализация по аналогии с авансами.

## 2. Заявки на материалы из процесса работ (для склада)

### 2.1 Создание заявки из процесса работ

**Эндпоинт (вызывается из UI процесса работ):**
- `POST /work-entries/{id}/warehouse-requests/`

**Тело:**

```json
{
  "warehouse": "uuid-склада",
  "items": [
    {"nomenclature": "uuid-позиции", "quantity": "10.00", "unit": "шт"},
    {"nomenclature": "uuid-позиции-2", "quantity": "5.00", "unit": "м2"}
  ],
  "comment": "Материалы для отделочных работ"
}
```

**Логика:**
- создаётся заявка на материалы, привязанная к:
  - процессу работ (`work_entry`),
  - складу (`warehouse`),
  - списку номенклатур и количеств;
- начальный статус заявки: `pending` (ожидает обработки склада).

### 2.2 Просмотр заявок со стороны склада

Чтобы склад видел все запросы:

**Список заявок:**
- `GET /work-entries/warehouse-requests/`

**Детали заявки:**
- `GET /work-entries/warehouse-requests/{id}/`

**Фильтры списка (query):**
- `warehouse`
- `status` (`pending`, `approved`, `rejected`, `partially_approved`, `completed`)
- `work_entry`
- `residential_complex`

**Ответ по заявке (пример):**
- `id`
- `work_entry` — id и краткая информация о процессе работ.
- `warehouse`
- `items[]` — номенклатура, запрошенное количество, единица.
- `status`
- `created_at`, `updated_at`
- при необходимости: выданные количества и ссылки на движения склада.

Склад на основе этих данных принимает решение, какие позиции и в каких количествах выдавать.

## 3. Передача товаров по заявке в процесс работ (движение склада)

### 3.1 Передача по заявке: `/warehouse-movements/transfer-to-work-entry/`

**Эндпоинт (вызывается со стороны склада):**
- `POST /warehouse-movements/transfer-to-work-entry/`

**Тело (по заявке):**

```json
{
  "warehouse_request": "uuid-заявки-на-материалы",
  "warehouse": "uuid-склада",
  "items": [
    {"nomenclature": "uuid-позиции", "quantity": "8.00", "unit": "шт"},
    {"nomenclature": "uuid-позиции-2", "quantity": "5.00", "unit": "м2"}
  ],
  "comment": "Выдача материалов по заявке",
  "files": []
}
```

**Поля:**
- `warehouse_request` — заявка, по которой выполняется выдача.
- `warehouse` — склад, с которого выдаются материалы (может подтягиваться из заявки).
- `items[]` — фактически выдаваемые количества по каждой номенклатуре (не больше запрошенных).
- `comment` — комментарий склада.
- `files` — опционально: список файлов для прикрепления к движению (акты выдачи и т.п.).

**Логика backend:**
1. Найти заявку по `warehouse_request`, получить связанный `work_entry` и список запрошенных позиций.
2. Для каждой позиции в `items`:
   - проверить, что выдаваемое количество не превышает оставшийся к выдаче объём по заявке;
   - создать движение склада (`warehouse-movement`) на **списание** товара с указанного склада:
     - `movement_type = "transfer_to_work_entry"` (или аналог);
     - ссылка на `warehouse`, `work_entry`, `warehouse_request`;
     - номенклатура, количество, единица, комментарий.
3. Обновить агрегаты по заявке:
   - сколько выдано по каждой позиции;
   - статус заявки:
     - `partially_approved`, если часть позиций/количеств ещё не выдана;
     - `completed`, если всё выдано;
     - при полном отказе/ошибке — `rejected` (через отдельный маршрут).
4. Прикрепить файлы к движениям склада (по `warehouse-movements/{id}/files/`), если переданы.
5. Вернуть созданные движения и обновлённое состояние заявки.

Таким образом:
- все перемещения материалов по заявке фиксируются в **журнале движений склада**;
- у процесса работ есть полная история: какие материалы, в каком объёме и когда были выданы;
- склад видит историю по каждой заявке и каждой номенклатуре.

### 3.2 Вариант без заявки

Для разовой передачи в процесс работ без предварительной заявки:

```json
{
  "work_entry": "uuid-процесса-работ",
  "warehouse": "uuid-склада",
  "items": [
    {"nomenclature": "uuid-позиции", "quantity": "3.00", "unit": "шт"}
  ],
  "comment": "Разовая выдача материалов",
  "files": []
}
```

- В этом случае поле `warehouse_request` опускается, движение привязывается напрямую к `work_entry`;
- остальные правила (списание со склада, фиксация движения, файлы) остаются теми же.
