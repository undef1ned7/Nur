# Маркет-касса: несколько корзин, скан и смешанная оплата

Спецификация для бэкенда и контракт с фронтом (`/crm/market/cashier`).

---

## 1. `POST /main/pos/sales/start/`

Единая точка: открытие смены, возврат к существующей корзине, создание новой, переключение.

### 1.1 Тело запроса

```json
{
  "shift": "uuid-смены",
  "order_discount_total": 0,
  "is_wholesale": false,
  "is_new": false,
  "sale_id": "uuid-опционально"
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `shift` | uuid | **Обязательно** при работе кассы — открытая смена. |
| `order_discount_total` | number | Скидка на заказ (как сейчас). |
| `is_wholesale` | bool | Розница / опт. |
| `is_new` | bool | **`true`** — всегда создать **новую** открытую корзину (кнопка «Новая» на фронте). **`false` или поле отсутствует** — не создавать новую: вернуть текущую открытую корзину смены или создать **первую (основную)**, если открытых нет. |
| `sale_id` | uuid | Опционально. Активировать указанную открытую продажу и вернуть её в `sale` (при переключении вкладки). |

**Алиас на фронте:** в коде допускается `isNew` → уходит как `is_new`.

### 1.2 Поведение `is_new`

| `is_new` | Действие |
|----------|----------|
| `true` | Новая POS-продажа (`status=open`), предыдущие open продажи смены **не закрываются**. |
| `false` / нет | Если есть open продажа (основная или последняя активная) — вернуть её; иначе создать основную. |

### 1.3 Ответ (обязательный контракт)

**Всегда** возвращать массив корзин и активную продажу:

```json
{
  "sale": {
    "id": "uuid",
    "shift": "uuid",
    "status": "open",
    "is_default": true,
    "items": [],
    "total": "0.00",
    "order_discount_total": "0.00",
    "is_wholesale": false
  },
  "active_sale_id": "uuid",
  "carts": [
    {
      "id": "uuid",
      "is_default": true,
      "label": "Основная",
      "items_count": 0,
      "total": "0.00",
      "status": "open"
    },
    {
      "id": "uuid-2",
      "is_default": false,
      "label": "Корзина 2",
      "items_count": 3,
      "total": "1250.00",
      "status": "open"
    }
  ]
}
```

| Поле | Описание |
|------|----------|
| `sale` | Полный объект **активной** продажи (с `items`), с которой работает кассир. |
| `active_sale_id` | UUID активной корзины (= `sale.id`). |
| `carts` | **Все** открытые корзины смены для вкладок UI. |

Поля элемента `carts[]`:

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | uuid | ID продажи (то же, что `sale_id`). |
| `is_default` | bool | Ровно одна **основная** корзина на смену. |
| `label` | string | Подпись вкладки («Основная», «Корзина 2», …). |
| `items_count` | int | Число позиций (для бейджа). |
| `total` | decimal string | Сумма корзины. |
| `status` | string | `open` |

### 1.4 Обратная совместимость

Если бэкенд пока отдаёт **только объект продажи** без `carts`, фронт строит одну вкладку «Основная» из этого объекта. После выката массива `carts` UI автоматически покажет все вкладки.

### 1.5 Когда фронт вызывает `start`

| Сценарий | Тело |
|----------|------|
| Открытие кассы / смены | `{ shift, is_wholesale }` без `is_new` |
| Кнопка «Новая» | `{ shift, is_new: true }` |
| После оплаты | `{ shift }` без `is_new` — обновить список `carts` |
| Переключение вкладки | `GET /sales/{id}/` + `{ shift, sale_id }` |
| Обновление после скана/правок | `{ shift }` или `GET` активной продажи |

## 2. `POST /main/pos/sales/{id}/scan/`

### 2.1 Тело

```json
{
  "barcode": "0693123456789",
  "sale_id": "uuid-опционально"
}
```

| Поле | Описание |
|------|----------|
| `barcode` | Штрих-код (строка, ведущие нули сохранять). |
| `sale_id` | **Опционально.** В какую корзину добавить товар. |

### 2.2 Логика на бэке

1. Если в теле передан `sale_id` — добавить в **эту** открытую продажу (должна принадлежать смене).
2. Если `sale_id` **нет** — добавить в **основную** корзину смены (`is_default=true`).
3. `{id}` в URL — продажа для маршрута; рекомендуется совпадать с `sale_id` или с основной, если `sale_id` не передан.

### 2.3 Ответ (рекомендуется)

Как минимум — обновлённая активная продажа. Желательно тот же формат, что и `start`:

```json
{
  "sale": { "id": "...", "items": [ ... ] },
  "carts": [ ... ],
  "active_sale_id": "..."
}
```

Иначе фронт обновляет только `sale` из тела ответа.

### 2.4 Фронт

- Скан с активной вкладки: в тело уходит `sale_id` активной корзины.
- Скан без `sale_id` (интеграция) → бэкенд кладёт в **основную**.
- Query-параметр страницы `?sale_id=` — фронт подставляет его в `sale_id` при скане.

## 3. Смешанная оплата (онлайн + перевод)

Без изменений к разделу 1–2; см. предыдущую версию документа.

### 3.1 Checkout (целевой контракт)

```json
POST /main/pos/sales/{id}/checkout/
{
  "print_receipt": true,
  "client_id": null,
  "payments": [
    { "method": "mbank", "amount": "1000.00" },
    { "method": "transfer", "amount": "500.00" }
  ]
}
```

Пока `payments` нет — фронт: один `payment_method` + два `addCashFlows` (временное решение).

---

## 4. Ограничения

- Лимит open-корзин на смену (рекомендация: 10).
- При закрытии смены — политика для всех `open` (cancel / force checkout).
- Основная корзина: `is_default=true`, одна на смену.

## 6. Чеклист для бэкенда

- [ ] `POST /start/` принимает `is_new`, без него — существующая или первая корзина.
- [ ] Ответ `start` **всегда** содержит `carts[]`, `sale`, `active_sale_id`.
- [ ] `POST .../scan/` принимает опциональный `sale_id`; без него — основная корзина.
- [ ] (Опционально) `scan` и `checkout` возвращают обновлённый `carts[]`.
- [ ] (Опционально) `checkout.payments[]` для смешанной оплаты.
