# Кафе: фильтр аналитики по времени (`time_from` / `time_to`)

Касается страницы `/crm/cafe/analytics`. Фронт уже отправляет новые параметры —
нужно, чтобы бэкенд начал их учитывать.

## Что отправляет фронт

Ко всем запросам аналитики кафе, где сейчас уходят `date_from` / `date_to`
(в т.ч. `GET /api/cafe/analytics/...`, `menu/all/`, экспорт), добавляются
**опциональные** параметры:

| Параметр | Формат | Описание |
|----------|--------|----------|
| `time_from` | `HH:MM` (24ч) | Время начала внутри периода (включительно) |
| `time_to` | `HH:MM` (24ч) | Время конца внутри периода (включительно) |

Пример: `?date_from=2026-07-01&date_to=2026-07-07&time_from=08:00&time_to=16:00`.

## Семантика

- Параметры **необязательны**. Без них поведение прежнее (фильтр только по датам).
- `time_from`/`time_to` сужают выборку **внутри каждого дня периода**:
  «заказы за 1–7 июля, созданные/оплаченные с 08:00 до 16:00 каждого дня».
  Это нужно для анализа по сменам (утро/вечер), часам пик и т.п.
- Если задан только `time_from` — от этого времени до конца дня; только `time_to` —
  с начала дня до этого времени.
- `time_from > time_to` (например 22:00–06:00) — «ночное окно» через полночь;
  если сложно — допустимо вернуть 400 с понятным сообщением, фронт покажет ошибку.
- Время применяется к тому же полю-основе, что и даты (`basis`, обычно `paid_at`
  или `created_at`), в часовом поясе компании.
- Должно влиять на все блоки ответа: KPI/summary, разбивки по меню/категориям,
  повара/официанты, динамика выручки, расходы, экспорт.

## Валидация

- Неверный формат времени → игнорировать параметр или 400 (согласованно).
- Неизвестные параметры не должны ломать запрос (обратная совместимость).

## Чек-лист

- [ ] Все эндпоинты аналитики кафе принимают `time_from`/`time_to` (HH:MM).
- [ ] Фильтр по времени применяется внутри каждого дня периода к полю-основе.
- [ ] Учитывается часовой пояс компании.
- [ ] Экспорт и `menu/all/` учитывают те же параметры.
- [ ] Без параметров — поведение без изменений.
