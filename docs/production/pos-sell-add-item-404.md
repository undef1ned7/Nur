# Производство — Продажа (POS): 404 «No Cart matches the given query» при добавлении товара

**Сектор:** Производство (используются общие POS-эндпоинты маркета; проблема касается и кассы Маркета).
**Страницы:** `/crm/production/sell/start?sale_id=<uuid>` (касса производства), аналогичный поток — `/crm/market/cashier` (касса маркета).
**Эндпоинты:** `POST /main/pos/sales/start/`, `POST /main/pos/sales/{sale_id}/add-item/`, `GET /main/pos/sales/{sale_id}/`, `POST /main/pos/sales/{sale_id}/checkout/`.
**Статус:** ⚠️ Требуется проверка/правка на бэкенде. Фронт уже добавил защитную логику (повторный `/start/` и ретрай), но она не помогает, если сам `/start/` возвращает «мёртвый» `sale_id`.

---

## 1. Проблема

При добавлении товара в корзину на странице продажи производства запрос падает:

```
POST /main/pos/sales/08e2fa37-ccd4-47a4-a49a-d9fb5d8d439b/add-item/
→ 404 Not Found
{"detail": "No Cart matches the given query."}
```

При этом `sale_id` фронт не выдумывает — он получен от бэкенда (из ответа `POST /main/pos/sales/start/` либо ранее выдан бэком и сохранён в URL страницы). То есть бэкенд в одном месте признаёт продажу существующей, а в другом (`add-item`, лукап модели **Cart**) — не находит её корзину.

Проблема воспроизводится повторно: после ошибки фронт заново вызывает `POST /start/` (без `sale_id`), получает активный `sale_id` и повторяет `add-item` — и снова получает тот же 404. Значит `/start/` возвращает продажу, у которой на бэке **нет валидной Cart** для `add-item`.

## 2. Последовательность запросов фронта (как воспроизводится)

Открытие страницы `/crm/production/sell/start`:

1. `GET /main/pos/shifts/…` — поиск своей открытой смены.
2. `POST /main/pos/sales/start/` с телом вида:
   ```json
   { "shift": "<shift_id>", "is_wholesale": false }
   ```
   Ответ используется как список корзин + активная продажа (`sale` / `carts[]` / `active_sale_id`). Полученный `sale_id` подставляется в URL страницы.
3. Если в URL уже был `sale_id` (возврат на страницу, вторая вкладка) и он отличается от активного:
   `GET /main/pos/sales/{sale_id}/`, затем `POST /start/` с `{"sale_id": "<sale_id>", "shift": …}`.
4. Добавление товара: `POST /main/pos/sales/{sale_id}/add-item/`:
   ```json
   { "product_id": "<uuid>", "quantity": 1 }
   ```
   → здесь и возникает 404 «No Cart matches the given query».

Защитная логика фронта (уже в коде, `SellStart.jsx`): при этом 404 фронт вызывает `POST /start/` **без** `sale_id`, берёт `active_sale_id` из ответа и повторяет `add-item` в него. Если бэк снова возвращает тот же/такой же «мёртвый» id — фронт бессилен, цикл упирается в тот же 404.

## 3. Что нужно проверить на бэкенде

Ошибка «No Cart matches the given query» — это `get_object_or_404(Cart, …)` во вьюхе `add-item`. Просим проверить согласованность между `/start/` и `add-item`:

1. **Инвариант:** любой `sale_id`, который бэкенд возвращает из `POST /main/pos/sales/start/` (как `sale.id`, `active_sale_id` или элемент `carts[]`), обязан приниматься `add-item` без 404. Если корзина оплачена (`checkout`), удалена или закрыта вместе со сменой — `/start/` не должен её возвращать, а должен создать/вернуть новую.
2. **Разные фильтры лукапа.** Вероятно, `add-item` ищет Cart с дополнительными условиями (например `status="active"` / `is_paid=False` / привязка к открытой смене / к текущему пользователю-кассиру / компании), которых нет в выдаче `/start/`. Типовые сценарии рассинхрона:
   - после `checkout` Cart помечается оплаченной/удаляется, но Sale остаётся и `/start/` снова возвращает её id;
   - Sale создана вообще без Cart (Cart создаётся лениво где-то ещё);
   - смена закрыта: Cart отвязана/закрыта, но `/start/` с новым `shift` возвращает старую продажу;
   - продажа принадлежит другому кассиру/смене, `add-item` фильтрует по `request.user`, а `/start/` — нет.
3. **Роль владельца.** На странице производства продажу ведёт **owner** (не отдельный кассир). Если `add-item` фильтрует Cart по кассиру смены, а смену открыл owner — лукап может не находить корзину именно для owner'а.
4. **Идемпотентность `/start/`.** Повторные вызовы `/start/` с одним `shift` не должны плодить продажи без корзин и не должны возвращать разные `sale_id` без причины.

## 4. Ожидаемое поведение

1. `POST /start/` возвращает только продажи, готовые принимать `add-item` (или создаёт новую, если активной нет).
2. `POST /sales/{id}/add-item/` для существующей, но уже неактивной продажи (оплачена/закрыта) возвращает **400 с внятным кодом** (например `{"code": "sale_closed", "detail": "…"}`), а не 404 «No Cart matches» — фронту нужно отличать «продажа закрыта» от «продажа не существует».
3. (Опционально, самое надёжное) `add-item` сам создаёт Cart, если у живой Sale её ещё нет.
