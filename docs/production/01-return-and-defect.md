# 01 — Раздельный учёт ВОЗВРАТОВ и БРАКА (модуль «Производство»)

> Статус фронтенда: **готов к подключению**. Фронт уже читает поля и эндпоинты,
> описанные ниже; при их отсутствии деградирует безопасно (показывает `0` /
> «Нет данных»), существующий функционал не ломается.

---

## 1. Бизнес-процесс

В модуле «Производство» агент получает товар от владельца (передача → приёмка),
продаёт его клиентам и может **вернуть** товар. Возврат бывает двух видов:

| Вид | Смысл | Что происходит со складом |
|-----|-------|---------------------------|
| **Возврат** (`is_defect = false`) | Качественный товар возвращается обратно | После приёмки владельцем товар **возвращается на склад готовой продукции** (qty_on_hand агента уменьшается, остаток склада растёт). |
| **Брак** (`is_defect = true`) | Товар испорчен / непригоден | Товар **списывается**, на склад НЕ возвращается. Фиксируется как потеря. |

Сейчас обе операции хранятся в одной сущности (`Return` с флагом `is_defect`),
но в **учёте, статистике и аналитике отображаются вместе** (одна карточка
«Брак (возвраты)»). Задача — **разделить их во всех отчётах**.

### Что требуется от бэкенда
1. Гарантировать, что флаг `is_defect` проставляется при создании возврата
   (он уже есть в `POST /main/returns/` и `POST /main/agents/me/returns/`).
2. Во **всех агрегатах аналитики** считать возврат и брак **раздельно**
   (новые поля `returns_*` и `defects_*`, новые ряды графиков, новые срезы по агентам).
3. Добавить две новые категории детализации (`card=returns`, `card=defects`).

Полная замена на две физически разные таблицы **не требуется** — достаточно
дискриминатора `is_defect`. Рекомендуемый подход — оставить одну таблицу
`Return` и разделять выборки по `is_defect` (минимально инвазивно, обратная
совместимость сохраняется). Раздел 6 описывает опциональную «жёсткую» модель.

## 2. Сущности и связи

```
Owner (компания/филиал)
  └── Agent (сотрудник-агент)
        └── Subreal (передача товара агенту)   [main/subreals/]
              └── Acceptance (приёмка)          [main/acceptances/]
              └── Return (возврат/брак)         [main/returns/]
                     ├── is_defect: bool        ← дискриминатор Возврат/Брак
                     ├── status: pending|approved|rejected
                     ├── product (FK)
                     ├── agent (FK)
                     ├── client (FK, nullable)
                     ├── qty, amount
                     └── returned_at, created_at
```

Связи:
- `Return.subreal → Subreal` (из какой партии возврат).
- `Return.product → Product` (готовая продукция).
- `Return.agent → Agent` (кто вернул).
- `Return.client → Client` (от какого клиента, опционально).

## 3. Изменения в БД

Таблица `return` (или текущее имя сущности возврата) — **новые/подтверждаемые поля**:

| Поле | Тип | Null | Default | Описание |
|------|-----|------|---------|----------|
| `is_defect` | boolean | no | `false` | **Уже должно быть.** `true` = брак (списание), `false` = возврат на склад. Индекс обязателен. |
| `status` | varchar(16) | no | `pending` | `pending` / `approved` / `rejected`. |
| `amount` | decimal(14,2) | yes | `0` | Денежная оценка возврата/брака (qty × цена). |
| `returned_at` | timestamp | yes | — | Дата операции (для динамики). |

**Индексы (для производительности аналитики):**
```sql
CREATE INDEX idx_return_is_defect       ON return (is_defect);
CREATE INDEX idx_return_status          ON return (status);
CREATE INDEX idx_return_agent_isdefect  ON return (agent_id, is_defect);
CREATE INDEX idx_return_returned_at     ON return (returned_at);
```

Миграция не удаляет данных. Если `is_defect` отсутствовал — добавить со
значением по умолчанию `false` и (опционально) бэкофилл по бизнес-правилу.

> **Важно для аналитики:** в агрегаты возврата/брака включать только записи со
> `status = 'approved'` (подтверждённые владельцем), чтобы pending-заявки не
> искажали отчёты. Это совпадает с тем, как фронт уже работает (`status=pending`
> для очереди, approved — для учёта).

## 4. API: аналитика производства (основной экран)

### 4.1. `GET /main/owners/analytics/`

Существующий эндпоинт сводной аналитики. **Расширяется** разделами по возврату и
браку. Параметры запроса — без изменений.

**Query params** (как сейчас):
| Параметр | Тип | Описание |
|----------|-----|----------|
| `period` | string | `day` / `week` / `month` / `custom` |
| `date` | date | при `period=day` |
| `date_from`, `date_to` | date | при кастомном диапазоне |
| `group_by` | string | `day` / `week` / `month` (гранулярность графиков) |
| `branch` | uuid | филиал (если применимо) |

**Response body — НОВЫЕ поля (добавить к существующему ответу):**

```jsonc
{
  "period": { "group_by": "day", "date_from": "2026-06-01", "date_to": "2026-06-24" },

  "summary": {
    // ... существующие поля без изменений ...
    "defective_items": 12,        // СТАРОЕ поле — оставить для обратной совместимости

    // НОВОЕ — раздельно:
    "returns_qty": 8,             // суммарное кол-во ВОЗВРАТОВ (is_defect=false, approved)
    "returns_amount": 1600.00,    // сумма возвратов, сом
    "defects_qty": 4,             // суммарное кол-во БРАКА (is_defect=true, approved)
    "defects_amount": 900.00      // сумма брака, сом
  },

  "charts": {
    // ... существующие графики без изменений ...

    // НОВОЕ — динамика (массивы точек по датам в гранулярности group_by):
    "returns_by_date": [
      { "date": "2026-06-20", "qty": 3, "amount": 600.00 },
      { "date": "2026-06-21", "qty": 5, "amount": 1000.00 }
    ],
    "defects_by_date": [
      { "date": "2026-06-20", "qty": 1, "amount": 200.00 },
      { "date": "2026-06-22", "qty": 3, "amount": 700.00 }
    ],

    // НОВОЕ — срезы по агентам (топ, отсортированный по qty desc):
    "top_agents_by_returns": [
      { "agent_id": "uuid", "agent_name": "Иванов И.", "qty": 5, "amount": 1000.00 }
    ],
    "top_agents_by_defects": [
      { "agent_id": "uuid", "agent_name": "Петров П.", "qty": 3, "amount": 700.00 }
    ]
  }
}
```

**Контракт полей (важно, фронт уже это читает):**
- Имена полей — **точно** как выше.
- `returns_by_date` / `defects_by_date`: `date` — ISO `YYYY-MM-DD` (или начало
  периода при `group_by=week/month`); `qty` — число; `amount` — число/строка.
- `top_agents_by_*`: `agent_name` обязателен (фронт показывает его), `agent_id`
  желателен (используется как React key), `qty` и `amount` — числа.
- Пустые срезы возвращать как `[]` (не `null`).

**Статусы:** `200 OK`. Ошибки — см. раздел 7.

## 5. API: детализация карточек (drill-down)

При клике на карточку «Возвраты» / «Брак» фронт открывает модалку и запрашивает:

### 5.1. `GET /main/analytics/cards/details/`

**Query params:**
| Параметр | Тип | Обяз. | Описание |
|----------|-----|-------|----------|
| `card` | string | да | **`returns`** или **`defects`** (новые значения) |
| `limit` | int | да | размер страницы (по умолчанию из фронта) |
| `offset` | int | да | смещение для пагинации |
| `branch` | uuid | нет | филиал |
| `period`, `date`, `date_from`, `date_to` | — | нет | для `returns`/`defects` период не передаётся (как у `defective_items`); агрегируется по текущему окну отчёта на стороне БЭ |

**Response body** (массив строк; фронт принимает `[]`, `{items:[]}`,
`{results:[]}` или `{rows:[]}`):

```jsonc
{
  "items": [
    {
      "product_id": "uuid",
      "product_name": "Стол дубовый",
      "agent_name": "Иванов И.",
      "client_name": "ООО Ромашка",   // nullable
      "qty": 3,
      "amount": 600.00,
      "returned_at": "2026-06-21T10:30:00Z"
    }
  ],
  "count": 42      // общее число строк (для пагинации), опционально
}
```

- `card=returns` → строки, где `is_defect=false`, `status=approved`.
- `card=defects` → строки, где `is_defect=true`, `status=approved`.
- Колонки и подписи на фронте уже сконфигурированы (`product_id`, `product_name`,
  `agent_name`, `client_name`, `qty`, `amount`, `returned_at`). Лишние поля
  игнорируются, отсутствующие отображаются как «—».

**Статусы:** `200 OK`; `400` при неизвестном `card`; см. раздел 7.

## 6. (Опционально) Полностью раздельные сущности

Если бизнес настаивает на **двух отдельных сущностях**, рекомендуемая модель:

- `Return` — только качественные возвраты (без `is_defect`).
- `Defect` — отдельная таблица брака (`product`, `agent`, `client`, `qty`,
  `amount`, `reason`, `status`, `created_at`, `returned_at`).

Эндпоинты-зеркала:
- `GET/POST /main/defects/`, `POST /main/defects/{id}/approve/`, `/reject/`.
- `GET/POST /main/agents/me/defects/`.

> Этот вариант дороже (миграция данных, дублирование логики приёмки/одобрения).
> **Рекомендация:** начать с дискриминатора `is_defect` (раздел 3) — фронт уже
> совместим. К отдельной таблице переходить только если появятся атрибуты,
> уникальные для брака (причина списания, фото-доказательство и т.п.).

## 7. Обработка ошибок

Единый формат (как во всём проекте — фронт показывает текст через `validateResErrors`):

```jsonc
// 400 Bad Request — неизвестная категория детализации
{ "detail": "Неизвестная категория карточки: foo" }

// 401 Unauthorized
{ "detail": "Учётные данные не были предоставлены." }

// 403 Forbidden — нет прав на аналитику
{ "detail": "Недостаточно прав для просмотра аналитики." }

// 422 / 400 — некорректный диапазон дат
{ "date_from": ["date_from не может быть позже date_to"] }
```

| Код | Когда |
|-----|-------|
| `200` | Успех |
| `400` | Неизвестный `card`, некорректные параметры |
| `401` | Нет авторизации |
| `403` | Нет прав (роль не owner / нет доступа к аналитике) |
| `500` | Внутренняя ошибка агрегации |

Все текстовые ошибки — на русском, человекочитаемые (фронт выводит их напрямую).

## 8. Права доступа

- `GET /main/owners/analytics/` и `GET /main/analytics/cards/details/` —
  **только владелец (`role = owner`)** и роли с правом просмотра аналитики
  (как сейчас). Логику прав **не менять** — только расширить ответ полями.
- Создание возврата/брака (`POST /main/agents/me/returns/`) — агент (как сейчас).
- Одобрение/отклонение (`/approve/`, `/reject/`) — владелец (как сейчас).

## 9. Последовательность действий (backend)

1. Убедиться, что `is_defect` есть в таблице `return` и проставляется при создании
   (`POST /main/returns/`, `POST /main/agents/me/returns/`). При отсутствии —
   миграция (раздел 3).
2. Добавить индексы из раздела 3.
3. В агрегаторе `GET /main/owners/analytics/`:
   - посчитать `summary.returns_qty/amount` = SUM по `is_defect=false, status=approved`;
   - посчитать `summary.defects_qty/amount` = SUM по `is_defect=true, status=approved`;
   - построить `charts.returns_by_date` и `charts.defects_by_date` (GROUP BY дате по `group_by`);
   - построить `charts.top_agents_by_returns` и `charts.top_agents_by_defects`
     (GROUP BY agent, ORDER BY qty DESC, LIMIT N).
   - **Сохранить** существующее поле `summary.defective_items` (обратная совместимость).
4. В `GET /main/analytics/cards/details/` добавить обработку `card=returns` и
   `card=defects` (фильтр по `is_defect`, проекция колонок из раздела 5).
5. Покрыть тестами: суммы `returns_* + defects_*` по qty должны сходиться с прежним
   совокупным `defective_items` (инвариант на период миграции).

## 10. Связь с фронтендом (что уже сделано)

| Фронт читает | Эндпоинт/поле |
|--------------|----------------|
| Карточка «Возвраты» (qty) | `summary.returns_qty` (фолбэк `regular_returns_qty`) |
| Карточка «Брак» (qty) | `summary.defects_qty` (фолбэк `defective_items`) |
| Drill-down «Возвраты» | `GET …/cards/details/?card=returns` |
| Drill-down «Брак» | `GET …/cards/details/?card=defects` |
| График «Динамика возвратов» | `charts.returns_by_date[]` |
| График «Динамика брака» | `charts.defects_by_date[]` |
| Таблица «Возврат по агентам» | `charts.top_agents_by_returns[]` |
| Таблица «Брак по агентам» | `charts.top_agents_by_defects[]` |

До появления этих полей фронт показывает `0` / «Нет данных» и не падает.
