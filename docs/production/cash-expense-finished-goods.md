# Производство — Касса: ГП из сырья не должна создавать расход

**Сектор:** Производство.
**Страницы:** `/crm/production/warehouse` (склад ГП и сырья), `/crm/kassa` (касса), `/crm/production/analytics`.
**Эндпоинты:** `POST /main/products/create-manual/`, `POST /construction/cashflows/` (или аналогичный эндпоинт создания операции кассы), item_make-списание сырья.
**Статус:** ⚠️ Требуется правка на бэкенде. Фронт уже ведёт себя правильно.

---

## 1. Проблема (жалоба клиента)

В кассу расходом попадают **и** закупка сырья, **и** произведённая из этого сырья готовая продукция (ГП). Получается двойной расход: деньги на сырьё уже потрачены при закупке, а при оприходовании ГП, сделанной из этого сырья, живых денег компания не тратит — списывается сырьё со склада, а не деньги из кассы.

**Правильная модель:**

| Операция | Денежный расход из кассы? |
|---|---|
| Закупка сырья | ✅ да, на сумму закупки |
| Производство ГП из сырья (по рецепту) | ❌ нет — себестоимость уже оплачена сырьём |
| Закупка готового товара (ГП без рецепта) | ✅ да, на сумму закупки |

## 2. Что уже делает фронт (не менять)

Аудит всех точек, где фронт сектора Производство создаёт операции кассы:

| Поток | Файл | Что создаёт |
|---|---|---|
| Закупка сырья (новая позиция) | `RawMaterialAddModal.jsx` | расход на сумму закупки (при оплате не в долг) |
| Докупка сырья (существующая позиция) | `AddRawMaterials.jsx` | расход «Закупка товара: …» (при оплате не в долг) |
| Создание ГП **с рецептом** из сырья | `FinishedGoodsAddModal.jsx` | **ничего** — расход сознательно НЕ создаётся (`!hasRecipeMaterials`) |
| Создание ГП **без рецепта** (закупленная готовая) | `FinishedGoodsAddModal.jsx` | расход на сумму закупки |
| Приёмка ГП от агента | `FinishedGoods.jsx` | приход «Приёмка: …» |

То есть лишний расход по произведённой ГП появляется **не с фронта** — его создаёт бэкенд (предположительно при `POST /main/products/create-manual/` или при списании сырья item_make).

## 3. Требуемое поведение бэкенда

1. При создании товара через `POST /main/products/create-manual/` бэкенд **не должен автоматически создавать расходную операцию кассы**, если товар производится из сырья, то есть выполняется любое из:
   - `recipe` непустой;
   - `item_make` непустой;
   - `calc_purchase_price_from_recipe = true`.
2. Списание сырья при производстве (item_make) — это **складская** операция (движение остатков), не денежная: операций в кассе она порождать не должна.
3. Для товара **без рецепта** бэкенд тоже не должен создавать расход сам: расход уже создаёт фронт отдельным запросом (иначе получится дубль). Если сейчас бэкенд создаёт — убрать; денежные операции кассы по этому потоку целиком на фронте.
4. Существующие ошибочные расходы у клиентов бэкенд может не чистить автоматически — достаточно прекратить создавать новые (при необходимости чистку обсудим отдельно).

## 4. Чек-лист приёмки

Перед проверкой зафиксировать текущее число операций в кассе.

| # | Действие | Ожидание |
|---|---|---|
| 1 | Закупить сырьё на 1000 сом (оплата наличными) | в кассе появился **один** расход на 1000 |
| 2 | Создать ГП с рецептом из этого сырья (10 шт) | в кассе **не появилось ни одной** новой операции; сырьё списалось со склада |
| 3 | Создать ГП без рецепта (закупка готовой) на 500 сом | в кассе появился **ровно один** расход на 500 (от фронта), не два |
| 4 | Создать ГП с рецептом при оплате «Долги» | операций в кассе нет, долг создан |
| 5 | Аналитика производства после шагов 1–3 | расходы за период = 1500 (1000 сырьё + 500 закупка ГП), без стоимости произведённой ГП |
