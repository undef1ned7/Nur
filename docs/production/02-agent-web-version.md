## 2. Уведомления (добавленная функция)

### 2.1. Используемое API (уже существует)

```
GET  /main/notifications/                 → { "results": [...], "count": N }
POST /main/notifications/mark-all-read/    → 200
GET  /main/notifications/{id}/             → объект уведомления (детали)
```

**Элемент уведомления (терпимый контракт; фронт читает альтернативы):**
```jsonc
{
  "id": "uuid",
  "title": "Передача принята",        // title | verb | subject
  "message": "Стол дубовый, 20 шт",   // message | description | text | body
  "is_read": false,                    // is_read | read
  "created_at": "2026-06-24T18:40:00Z" // created_at | created | date
}
```

Поведение фронта:
- бейдж с числом **непрочитанных** (`is_read=false`);
- поллинг каждые **60с** + перезагрузка при открытии;
- «Прочитать все» → `POST /main/notifications/mark-all-read/`;
- адаптивный dropdown (`width: min(360px, 92vw)` — без горизонтального скролла).

### 2.2. Какие события должны порождать уведомления агенту (требование к БЭ)

Бэкенд должен создавать уведомление пользователю-агенту при:

| Событие | Кому | Пример текста |
|---------|------|----------------|
| Назначена передача (`subreal`) | агенту | «Вам передан товар: {product}, {qty} шт» |
| Возврат одобрен/отклонён | агенту-инициатору | «Возврат {product} принят/отклонён» |
| Нехватка готовой продукции по заявке (см. **doc 06**) | ответственным/агенту | «Недостаточно {product}: запрошено {req}, доступно {avail}» |
| Долг клиента просрочен (опц.) | агенту | «Просрочен долг клиента {client}» |

> Достаточно записи в таблице `notification` с полями раздела 2.1 и адресатом
> `user_id` агента. Канал доставки — существующий `/main/notifications/`.

### 2.3. Изменения в БД (если ещё нет)

Таблица `notification` (вероятно уже существует). Требуемые поля:

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | uuid | PK |
| `user_id` | FK → user | адресат |
| `title` | varchar | заголовок |
| `message` | text | текст |
| `is_read` | boolean (default false) | прочитано |
| `created_at` | timestamp | дата |
| `kind` / `type` | varchar (опц.) | тип события (`transfer`, `return`, `shortage`, …) |
| `meta` | json (опц.) | данные события (product_id, qty, agent_id …) |

Индексы: `(user_id, is_read)`, `(user_id, created_at)`.

## 3. Контракт агентских эндпоинтов (для справки БЭ)

Эти эндпоинты уже используются веб-агентом — менять не нужно, перечислены для
полноты картины:

| Операция | Метод/URL |
|----------|-----------|
| Товары агента | `GET /main/agents/me/products/` |
| Обновление своих товаров | `PATCH /main/agents/me/products/` |
| Возврат (создать) | `POST /main/agents/me/returns/` |
| Мои возвраты | `GET /main/agents/me/returns/` |
| Передачи (subreals) | `GET/POST /main/subreals/` |
| Приёмка | `POST /main/accept/`, `/main/acceptances/` |
| Клиенты | `GET /main/clients/`, долги: `/main/clients/{id}/deals/` |
| Продажи (история) | агентские sales-эндпоинты (`/main/agents/me/...`) |
| Заказы/чекаут | агентская корзина → checkout |
| Уведомления | `GET /main/notifications/`, `POST /main/notifications/mark-all-read/` |

Права: агент видит/меняет **только свои** данные (фильтрация по `request.user`
на стороне БЭ — как сейчас). Логику прав не менять.

## 4. Адаптивность (frontend, проверено)

| Брейкпойнт | Поведение |
|------------|-----------|
| Desktop ≥1200px | таблицы; полный набор действий в шапке |
| Laptop 1024–1199 | таблицы/карточки (переключатель), сохраняется в localStorage |
| Tablet 768 | карточки; кнопки переносятся (`flex-wrap`) |
| Mobile ≤430 | карточки; колокольчик-dropdown `min(360px, 92vw)` — **нет горизонтального скролла**; кнопки full-width-friendly |

Существующие экраны агента уже адаптивны (режим table/cards, `useResize`,
`data-label` для мобильных таблиц). Добавленный колокольчик не вызывает
горизонтального скролла и закрывается по клику вне / по «×».

## 5. Обработка ошибок

- Ошибки уведомлений — через `validateResErrors` (баннер внутри dropdown),
  список не пропадает, приложение не падает.
- При `401/403` — стандартное сообщение; при пустом списке — «Уведомлений нет».

| Код | Когда |
|-----|-------|
| `200` | Успех |
| `401` | Нет авторизации |
| `403` | Нет прав |
| `500` | Внутренняя ошибка |

## 6. Последовательность действий (backend)

1. Убедиться, что `GET /main/notifications/` возвращает `{results, count}` с
   полями раздела 2.1 и фильтрует по текущему пользователю.
2. Реализовать/проверить `POST /main/notifications/mark-all-read/`.
3. Генерировать уведомления агенту на события раздела 2.2 (минимум — передача и
   решение по возврату; нехватка продукции — совместно с **doc 06**).
4. (Опц.) поля `kind`/`meta` для типизации и deep-link во фронте.
