# 04 — Расходы из КАССЫ в Аналитике «Производство»

> Статус фронтенда: **готов к подключению**. Компонент
> `ProductionExpensesFromCash` уже запрашивает расходы из существующего
> эндпоинта кассы и рендерит детальный список с фильтром/сортировкой/экспортом.
> Если бэкенд пока не поддерживает серверные фильтры — фронт фильтрует/сортирует
> на клиенте, ничего не ломается.

---

## 1. Бизнес-процесс

Расходы создаются **в модуле Касса** (например, через «Добавить операцию» →
`type = expense`, либо автоматически при оплате долга поставщику и т.п.).
Все они сохраняются в одной коллекции движений кассы (`cashflows`).

Задача: в разделе **Аналитика Производства → Расходы** показывать те же расходы
кассы **детальным списком** с полями:

| Колонка | Источник (поле движения кассы) |
|---------|--------------------------------|
| Дата | `created_at` |
| Сумма | `amount` |
| Категория | `category_title` / `category` |
| Комментарий | `name` / `comment` |
| Пользователь | `user_name` / `created_by_name` |
| Касса | `cashbox_name` / `department_name` |

Плюс: **фильтрация**, **сортировка**, **экспорт (CSV)**.

> Данные **автоматически подтягиваются из кассы** — отдельного хранилища
> «расходов производства» не создаётся. Это read-only представление поверх
> существующих движений кассы (`type = expense`).

---

## 2. Сущности и связи

```
CashBox (касса)                         [/construction/cashboxes/ или аналог]
  └── CashFlow (движение по кассе)       [/construction/cashflows/]
        ├── type: income | expense       ← фильтруем expense
        ├── amount: decimal
        ├── name / comment: text         ← комментарий операции
        ├── category (FK или title)      ← статья расхода
        ├── cashbox (FK) → CashBox
        ├── user / created_by (FK) → User  ← кто создал
        ├── created_at: timestamp
        └── source_business_operation_id ← откуда пришла операция (продажа/долг/ручная)
```

Связи:
- `CashFlow.cashbox → CashBox`
- `CashFlow.user → User`
- `CashFlow.category → ExpenseCategory` (если категории вынесены в справочник)

---

## 3. Изменения в БД

Новых таблиц **не требуется**. Нужно лишь убедиться, что в **ответе** эндпоинта
движений кассы присутствуют поля для отображения. Если каких-то нет — добавить
в сериализатор (не обязательно новые колонки, можно вычисляемые/`*_name`):

| Поле в ответе | Тип | Обяз. | Комментарий |
|---------------|-----|-------|-------------|
| `id` | uuid/int | да | ключ строки |
| `type` | string | да | `expense` / `income` |
| `amount` | decimal | да | сумма (положительная) |
| `created_at` | timestamp | да | дата/время операции |
| `name` (или `comment`) | string | да | комментарий |
| `category_title` (или `category`) | string | желат. | статья расхода; если нет — «—» |
| `cashbox_name` | string | желат. | имя кассы (читаемое) |
| `user_name` (или `created_by_name`) | string | желат. | ФИО/email автора |

> Фронт **толерантен** к именам: читает `amount|sum|value|total`,
> `type|kind|direction`, `category_title|category_name|category`,
> `comment|note|description|title|name`, `created_at|created|date|timestamp`,
> `user_name|created_by_name|user|created_by|cashier_name`,
> `cashbox_name|department_name|box_name|cashbox`. Достаточно вернуть любое из
> перечисленных — отсутствующее покажется как «—».

**Индексы (для фильтра по периоду/типу):**
```sql
CREATE INDEX idx_cashflow_type        ON cashflow (type);
CREATE INDEX idx_cashflow_created_at  ON cashflow (created_at);
CREATE INDEX idx_cashflow_cashbox     ON cashflow (cashbox_id);
```

## 4. API

### 4.1. `GET /construction/cashflows/` (существующий эндпоинт)

Используется **как есть**, расширяется поддержкой серверных фильтров. Фронт уже
шлёт эти query-параметры:

| Параметр | Тип | Обяз. | Описание |
|----------|-----|-------|----------|
| `type` | string | нет | `expense` — вернуть только расходы. **Рекомендуется поддержать.** |
| `date_from` | date (`YYYY-MM-DD`) | нет | начало периода (по `created_at`) |
| `date_to` | date (`YYYY-MM-DD`) | нет | конец периода |
| `cashbox` | uuid | нет | фильтр по конкретной кассе (уже поддерживается) |
| `branch` | uuid | нет | филиал (если применимо) |
| `ordering` | string | нет | сортировка, напр. `-created_at`, `amount` (опц., фронт умеет сортировать сам) |

**Response body** (массив или пагинированный объект — фронт принимает оба):

```jsonc
// Вариант А — массив
[
  {
    "id": "uuid",
    "type": "expense",
    "amount": 1500.00,
    "name": "Закуп упаковки",
    "category_title": "Материалы",
    "cashbox_name": "Основная касса",
    "user_name": "Иванов И.",
    "created_at": "2026-06-24T18:05:00Z"
  }
]

// Вариант Б — пагинация (фронт читает .results)
{ "count": 124, "next": "...", "previous": null, "results": [ /* как выше */ ] }
```

**Статусы:** `200 OK`. Ошибки — раздел 6.

> Если бэкенд **не** поддерживает `type=expense` / `date_from` / `date_to` —
> фронт всё равно отфильтрует расходы и период на клиенте. Но серверная
> фильтрация предпочтительнее (меньше трафика на больших объёмах).

### 4.2. Создание расхода (контекст, существующий)

`POST /construction/cashflows/`
```jsonc
{ "name": "Закуп упаковки", "amount": "1500.00", "type": "expense", "cashbox": "uuid",
  "source_business_operation_id": "Ручная операция" }
```
→ именно эти записи затем показываются в аналитике. Изменять создание **не нужно**.

## 5. Фильтрация / сортировка / экспорт

| Возможность | Где реализовано | Детали |
|-------------|------------------|--------|
| **Фильтр по периоду** | сервер (`date_from`/`date_to`) + дублируется фильтрами аналитики | период берётся из верхних фильтров страницы Аналитики |
| **Поиск** (категория/комментарий/пользователь/касса) | клиент | мгновенный, по подстроке |
| **Сортировка** (дата, сумма, категория, пользователь, касса) | клиент (по клику на заголовок) | сервер может дополнительно поддержать `ordering` |
| **Экспорт CSV** | клиент | UTF-8 + BOM (кириллица в Excel), колонки = видимым |
| **Итого** | клиент | сумма по отфильтрованным строкам |

Экспорт — **полностью на фронте** (Blob → download), серверный эндпоинт экспорта
не требуется. Если в будущем понадобится серверный экспорт больших объёмов —
добавить `GET /construction/cashflows/export/?type=expense&date_from&date_to`
(вернуть `text/csv`).

## 6. Обработка ошибок

Формат — общий по проекту (фронт показывает текст через `validateResErrors`):

```jsonc
{ "detail": "Учётные данные не были предоставлены." }      // 401
{ "detail": "Недостаточно прав." }                          // 403
{ "date_from": ["Некорректная дата."] }                     // 400
```

| Код | Когда |
|-----|-------|
| `200` | Успех |
| `400` | Некорректные параметры фильтра |
| `401` | Нет авторизации |
| `403` | Нет прав на просмотр кассы/аналитики |
| `500` | Внутренняя ошибка |

При ошибке фронт показывает баннер с текстом и пустую таблицу (не падает).

---

## 7. Права доступа

- `GET /construction/cashflows/` — доступ как у текущей Кассы/Аналитики
  (владелец и роли с доступом к кассе). **Логику прав не менять.**
- Аналитика Производства видна владельцу/ролям аналитики; компонент расходов
  наследует те же права (отдельной проверки не добавляем).

1. В `GET /construction/cashflows/` добавить поддержку query-параметров
   `type`, `date_from`, `date_to` (фильтр по `created_at`).

   `category_title`/`category`, `cashbox_name`, `user_name`/`created_by_name`,
   `type`.
3. Добавить индексы (раздел 3).
4. (Опц.) Поддержать `ordering` (`-created_at`, `amount`, `category_title` …).
5. (Опц., на будущее) серверный CSV-экспорт.
