# Производство — Отчёт «Произведено за период» (день/ночь)

**Страница:** `/crm/production/warehouse`, вкладка «Производство» (фронт: `src/Components/Sectors/Production/Warehouse/ProductionReportTab.jsx`).
**Эндпоинт:** `GET /main/production/report/` — **новый**.
**Статус:** фронт готов и ждёт эндпоинт; пока API отвечает 404, вкладка показывает заглушку «после обновления сервера».

---

## 1. Задача

Клиент хочет видеть, **что и сколько произведено** за неделю/месяц (произвольный период), со списком записей и с разбивкой по сменам **день/ночь**.

Границы смен (местное время компании, Asia/Bishkek, UTC+6):

- **день** — с 08:00:00 до 19:59:59;
- **ночь** — с 20:00:00 до 07:59:59 (ночная смена, начавшаяся в 20:00, относится к дате своего начала).

## 2. Источник данных

Записью «производство» считается каждое событие изготовления ГП из сырья:

1. Создание товара через `POST /main/products/create-manual/` с непустым `recipe`/`item_make` — количество `quantity` из запроса.
2. Любые будущие потоки «произвести ещё» по существующему рецепту.

У каждой записи должны сохраняться: дата-время производства, товар, количество, единица, себестоимость (сумма списанного сырья), автор (пользователь, оформивший производство). Закупка готового товара без рецепта производством **не считается** и в отчёт не попадает.

## 3. Контракт API

```
GET /main/production/report/?date_from=2026-07-01&date_to=2026-07-14&shift=day&search=хлеб&limit=25&offset=0
```

### Параметры

| Параметр | Тип | Описание |
|---|---|---|
| `date_from` | date `YYYY-MM-DD` | Начало периода включительно (местное время, с 00:00:00) |
| `date_to` | date `YYYY-MM-DD` | Конец периода включительно (до 23:59:59) |
| `shift` | `day` \| `night` | Только дневные / только ночные производства; без параметра — все |
| `search` | string | Поиск по названию товара |
| `limit`, `offset` | int | Пагинация (как в `/main/stock-movements/`) |

### Ответ

```json
{
  "count": 132,
  "next": "…",
  "previous": null,
  "summary": {
    "total_quantity": "1540.000",
    "day_quantity": "980.000",
    "night_quantity": "560.000",
    "total_cost": "125400.50"
  },
  "results": [
    {
      "id": "uuid",
      "product": "uuid",
      "product_name": "Хлеб формовой",
      "quantity": "120.000",
      "unit": "шт.",
      "cost_total": "3600.00",
      "produced_by": "uuid",
      "produced_by_name": "Айбек У.",
      "produced_at": "2026-07-13T21:15:00",
      "shift": "night"
    }
  ]
}
```

Правила:

1. **`summary` считается по всему периоду с учётом фильтров** (`shift`, `search`), а не по текущей странице. Фронт показывает эти четыре числа в карточках сводки.
2. `shift` в каждой записи бэкенд проставляет сам по правилам из раздела 1 (по местному времени!). Фронт умеет вычислить смену из `produced_at` как фолбэк, но серверное поле — источник истины (важно для фильтра `shift`).
3. `produced_at` — ISO `YYYY-MM-DDTHH:MM:SS`, местное время компании (та же договорённость, что в `docs/market/analytics.md`).
4. Денежные значения и количества — строки Decimal, как в остальных API.
5. Сортировка по умолчанию — `produced_at` по убыванию (новые сверху).
6. Пока записей нет (старые компании) — пустой `results` и нулевой `summary`, не 500.
7. Невалидные `date_from`/`date_to`/`shift` — HTTP 400 с `detail`.

## 4. Чек-лист приёмки

Данные: три производства 13.07 — 09:00 (100 шт), 15:00 (50 шт), 21:30 (60 шт); одна закупка готового товара без рецепта 13.07 в 10:00.

| # | Запрос | Ожидание |
|---|---|---|
| 1 | `date_from=2026-07-13&date_to=2026-07-13` | `count=3` (закупка готового не попала), `summary.total_quantity=210`, `day=150`, `night=60` |
| 2 | `…&shift=day` | 2 записи (09:00 и 15:00), `summary` только по ним |
| 3 | `…&shift=night` | 1 запись (21:30), `shift="night"` в записи |
| 4 | Производство в 07:30 | `shift="night"` |
| 5 | Производство в 08:00 ровно | `shift="day"` |
| 6 | `search=<название товара>` | фильтр работает вместе с датами и сменой |
| 7 | Период без производств | `results=[]`, `summary` нули, HTTP 200 |
| 8 | `shift=abc` | HTTP 400 с `detail` |
