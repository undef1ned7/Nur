# 06 — Уведомление о НЕХВАТКЕ готовой продукции

> Статус фронтенда: **готов к подключению**. Нехватка уже блокируется в UI с
> понятной ошибкой и отправляет событие нехватки. Отображение уведомлений
> работает через существующий колокольчик (`AgentNotificationsBell` →
> `GET /main/notifications/`). При отсутствии нового эндпоинта отправка события —
> best-effort (тихо логируется), основной поток не ломается.

---

## 1. Бизнес-процесс

Когда агент запрашивает готовую продукцию, а на складе её недостаточно:

```
Запрос:      100 шт
На складе:    60 шт
────────────────────
Результат:   ВЫДАЧА ЗАПРЕЩЕНА
```

Система должна:
1. **Запретить выдачу** (нельзя передать больше, чем есть).
2. Показать **понятную ошибку** агенту/оператору (запрошено N, доступно M).
3. **Уведомить ответственных сотрудников** (владелец/менеджеры склада).
4. Создать запись в **журнале событий**.
5. **Отобразить уведомление в интерфейсе** (колокольчик уведомлений).

Уведомление содержит: **товар, запрошенное количество, доступное количество,
агента, дату и время**.

---

## 2. Сущности и связи

```
StockShortageEvent (событие нехватки)          [/main/stock-shortages/]  (НОВАЯ)
   ├── product (FK) → Product
   ├── requested_qty
   ├── available_qty
   ├── agent (FK) → Agent (nullable)
   ├── source: transfer | request | sale
   ├── created_by (FK) → User       (кто инициировал выдачу)
   ├── created_at
   └──> порождает Notification(s) для ответственных

Notification (существующая)                     [/main/notifications/]
   ├── recipient (FK) → User
   ├── type: stock_shortage | …
   ├── title, message
   ├── is_read
   ├── data: { product, requested, available, agent, ... }
   └── created_at
```

Связи:
- `StockShortageEvent.product → Product`, `.agent → Agent`, `.created_by → User`.
- Событие нехватки порождает `Notification` для каждого ответственного.

«Ответственные сотрудники» = владелец компании/филиала + роли с правом
управления складом Производства (определяется на бэкенде по компании/филиалу
товара).

## 3. Изменения в БД

### `stock_shortage_event` (новая)
| Поле | Тип | Null | Описание |
|------|-----|------|----------|
| `id` | uuid | no | PK |
| `product_id` | FK → product | no | товар |
| `requested_qty` | decimal(14,3) | no | запрошено |
| `available_qty` | decimal(14,3) | no | доступно на складе на момент запроса |
| `agent_id` | FK → agent | yes | агент, запросивший товар |
| `source` | varchar(16) | no | `transfer` / `request` / `sale` |
| `created_by_id` | FK → user | no | инициатор операции |
| `created_at` | timestamp | no | дата/время |

### `notification` (существующая) — убедиться, что поддерживает:
| Поле | Тип | Описание |
|------|-----|----------|
| `type` | varchar | добавить значение `stock_shortage` |
| `data` | json | произвольные данные (товар/кол-ва/агент) для UI |

**Индексы:** `stock_shortage_event(created_at)`, `stock_shortage_event(product_id)`,
`notification(recipient_id, is_read)`.

## 4. API

### 4.1. Сообщить о нехватке (создаёт событие + уведомления)

`POST /main/stock-shortages/`

**Request body** (то, что шлёт фронт):
```jsonc
{
  "product": "uuid",
  "product_name": "Стол дубовый",
  "requested_qty": 100,
  "available_qty": 60,
  "agent": "uuid | null",
  "agent_name": "Иванов И.",
  "source": "transfer"          // transfer | request | sale
}
```

**Логика:**
1. Создать `StockShortageEvent` (журнал событий).
2. Определить ответственных (владелец/менеджеры склада товара).
3. Создать `Notification(type=stock_shortage)` каждому ответственному, с
   `title` и `message`, и `data` = деталями нехватки.
4. Вернуть созданное событие.

**Response `201 Created`:**
```jsonc
{
  "id": "uuid",
  "product_name": "Стол дубовый",
  "requested_qty": 100,
  "available_qty": 60,
  "agent_name": "Иванов И.",
  "source": "transfer",
  "created_at": "2026-06-24T20:10:00Z",
  "notified": 2                  // скольким сотрудникам отправлено (опц.)
}
```

**Пример уведомления** (то, что увидят в колокольчике):
```
Заголовок: «Нехватка товара на складе»
Текст: «Агент Иванов И. запросил "Стол дубовый": 100 шт,
        доступно 60 шт. 24.06.2026 20:10»
```

### 4.2. Серверная защита на выдаче (обязательно)

Эндпоинты выдачи/передачи **обязаны** отклонять выдачу при нехватке (defense in
depth — клиентская проверка не единственная):

`POST /main/subreals/` и `POST /main/subreals/bulk/` (передача агенту):
- если `qty_transferred > product.quantity` → `400` и (рекомендуется) создать
  `StockShortageEvent` + уведомления автоматически.

```jsonc
// 400 Bad Request
{
  "detail": "Недостаточно товара \"Стол дубовый\". Запрошено 100, доступно 60.",
  "shortage": { "product": "uuid", "requested_qty": 100, "available_qty": 60 }
}
```

> Так нехватка фиксируется и при прямом вызове API, даже минуя UI-проверку.

### 4.3. Чтение уведомлений (существующее)

- `GET /main/notifications/?is_read=false` → список (фронт-колокольчик).
- `POST /main/notifications/mark-all-read/` → отметить все прочитанными.
- `GET /main/notifications/{id}/` → детали.

Фронт уже умеет это (`AgentNotificationsBell`), читает `title|verb|subject`,
`message|description|text|body`, `created_at`, `is_read`.

## 5. Обработка ошибок

```jsonc
{ "requested_qty": ["Обязательное поле."] }            // 400
{ "detail": "Товар не найден." }                        // 404
{ "detail": "Недостаточно прав." }                      // 403
```

| Код | Когда |
|-----|-------|
| `201` | Событие создано, уведомления отправлены |
| `400` | Некорректное тело |
| `403` | Нет прав |
| `404` | Товар не найден |
| `500` | Внутренняя ошибка |

Отправка события с фронта — **best-effort**: ошибка этого запроса не мешает
пользователю (нехватка уже заблокирована и показана). Запрещено использовать
`console.log` как единственный канал — пользователю всегда показывается понятная
ошибка о нехватке (через `validationError`/`validateResErrors`).

---

## 6. Права доступа

- `POST /main/stock-shortages/` — авторизованные пользователи Производства
  (тот, кто инициирует выдачу). Получатели уведомлений (ответственные)
  определяются на бэкенде.
- Серверная защита на `subreals` — без изменения прав, только добавить проверку
  остатка и фиксацию события.
- Чтение уведомлений — текущий пользователь видит свои (как сейчас).

---

## 7. Последовательность действий (backend)

1. Создать таблицу `stock_shortage_event` (раздел 3) + индексы.
2. Добавить тип уведомления `stock_shortage` (+ поле `data` в notification, если нет).
3. Реализовать `POST /main/stock-shortages/` (событие + уведомления ответственным).
4. Добавить серверную проверку остатка в `POST /main/subreals/` и
   `/main/subreals/bulk/` → `400` + автосоздание события нехватки.
5. Определить правило «ответственные сотрудники» (владелец/менеджеры склада).
6. Тесты:
   - выдача `qty > stock` отклоняется (`400`);
   - создаётся `StockShortageEvent` и ≥1 `Notification`;
   - уведомление содержит товар/запрошено/доступно/агента/дату.

## 8. Связь с фронтендом (что уже сделано)

| Фронт | Деталь |
|-------|--------|
| Блокировка выдачи | `FinishedGoods → TransferProductModal.validateForm`: при `qty > доступно` выдача не отправляется |
| Понятная ошибка | «Недостаточно товара "X". Запрошено: N, доступно: M» |
| Отправка события | `reportStockShortage()` → `POST /main/stock-shortages/` (best-effort) — файл `Production/stockShortage.js` |
| Уведомление в UI | существующий `AgentNotificationsBell` (`GET /main/notifications/`) |
| Переиспользование | `reportStockShortage` можно вызвать в любой точке выдачи/запроса (`source: transfer/request/sale`) |

Поля события (товар, запрошенное/доступное кол-во, агент, дата/время) полностью
покрыты телом запроса; дату/время и получателей проставляет бэкенд.
