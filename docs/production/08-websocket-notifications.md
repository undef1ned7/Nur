# 08 — Real-time уведомления через WebSocket

> Статус фронтенда: **готов к подключению**. Хук `useNotificationsSocket`
> подключается к WS, кладёт уведомления в общий redux-слайс `notification`,
> обновляет колокольчик в шапке в реальном времени, играет звук, переподключается
> и защищён от дублей. REST (история/прочтение) — существующий, расширен полями
> пагинации и `unread_count`.

---

## 1. Бизнес-процесс

Любое значимое событие в системе порождает уведомление, которое должно
**мгновенно** появляться у тех пользователей, кому оно положено по бизнес-логике
(компания, роль, права, подразделение, склад, связь с агентом и т.д.) — без
перезагрузки страницы и без поллинга.

Канал доставки: **WebSocket** + резервный REST (история, при первом открытии).

---

## 2. WebSocket-сервер

### 2.1. Эндпоинт и авторизация
```
WS  /ws/notifications/?token=<ACCESS_JWT>
```
- Токен берётся из query-параметра `token` (текущий `accessToken`).
- На подключении сервер валидирует JWT. Невалидный/протухший → закрыть соединение
  кодом **4401** (фронт обновит токен через `/users/auth/refresh/` и переподключится).
- Технология — Django Channels (проект уже использует `/ws/...`, напр.
  `/ws/consalting/funnel/`), channel layer на Redis для масштабирования.

### 2.2. После подключения — определить контекст и подписать на группы
Из пользователя по токену определить и подписать соединение на группы (channel
groups), а НЕ слать всем подряд:
- `company:<company_id>` — общая компания;
- `branch:<branch_id>` / `department:<dept_id>` — подразделение;
- `warehouse:<warehouse_id>` — склад;
- `role:<role>` — роль (owner/manager/agent/…);
- `user:<user_id>` — личные уведомления;
- `agent:<agent_id>` — если пользователь-агент.

Маршрутизация события = публикация в нужные группы. Широковещание (broadcast на
всех) — запрещено; всегда минимально необходимый набор групп.

### 2.3. Сообщения сервер→клиент
```jsonc
// приветствие (опц.)
{ "type": "connection_established", "user_id": "uuid" }

// новое уведомление
{
  "type": "notification",
  "data": {
    "id": "uuid",
    "type": "order_created",          // см. п.4
    "title": "Новый заказ",
    "message": "Заказ №1270 на 100 сом",
    "created_at": "2026-06-24T20:30:00Z",
    "url": "/crm/market/cashier?sale_id=…",   // ссылка для перехода
    "level": "info",                  // info | success | warning | high | critical
    "is_read": false,
    "actor_name": "Иванов И."          // инициатор действия
  }
}

// (опц.) точное число непрочитанных
{ "type": "unread_count", "count": 7 }

// ответ на ping
{ "type": "pong" }
```

### 2.4. Сообщения клиент→сервер
```jsonc
{ "action": "ping" }     // keep-alive, фронт шлёт каждые 25с
```

> Сервер должен переживать масштабирование без принудительных переподключений
> клиентов: состояние групп — в channel layer (Redis), сообщения публикуются
> через `group_send`. Клиент переподключается только при разрыве.

## 3. Структура уведомления (модель)

`notification` (расширить существующую):
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | uuid | идентификатор |
| `type` | varchar | тип события (п.4) |
| `title` | varchar | заголовок |
| `message` | text | описание |
| `url` | varchar | ссылка для перехода (nullable) |
| `level` | varchar | важность: `info`/`success`/`warning`/`high`/`critical` |
| `is_read` | bool | статус прочтения |
| `actor_id` | FK→user | инициатор действия (nullable) |
| `recipient_id` | FK→user | получатель |
| `company_id`, `branch_id` | FK | контекст для маршрутизации |
| `data` | json | произвольные данные для UI |
| `created_at` | timestamp | дата |

Индексы: `notification(recipient_id, is_read)`, `notification(created_at desc)`.

## 5. REST API (история / прочтение) — существующее, расширить

### 5.1. `GET /main/notifications/`
**Query:** `limit`, `offset` (пагинация / lazy load).
**Response:**
```jsonc
{
  "count": 132,
  "unread_count": 7,            // ← ДОБАВИТЬ: для точного счётчика в колокольчике
  "results": [ /* объекты уведомлений (п.3) */ ]
}
```

### 5.2. `POST /main/notifications/mark-all-read/`
Отметить все прочитанными. → `200`.

### 5.3. `POST /main/notifications/{id}/read/`  ← НОВЫЙ
Отметить ОДНО прочитанным. → `200 { id, is_read: true }`.

### 5.4. `GET /main/notifications/{id}/`
Детали одного уведомления.

**Ошибки:** `401` (нет/протух токен), `403` (нет прав), `404` (не найдено),
`400` (некорректные параметры). Тексты — на русском (фронт — `validateResErrors`).

## 6. Права доступа / маршрутизация

- WS-соединение — только авторизованный пользователь (валидный JWT).
- Получатели события определяются на сервере по бизнес-логике (компания/роль/
  склад/агент). Пользователь получает только свои группы.
- REST: пользователь видит/читает только свои уведомления.
- Менять существующие права **не нужно** — добавить только маршрутизацию событий
  по группам.

---

## 7. Последовательность действий (backend)

1. Поднять Channels-consumer на `/ws/notifications/` с JWT-аутентификацией
   (закрытие 4401 при невалидном токене).
2. На `connect`: определить компанию/роль/подразделение/склад/агента → подписать
   соединение на соответствующие группы; реализовать `ping`→`pong`.
3. Расширить модель `notification` полями п.3; миграция.
4. При каждом событии (п.4) создавать `Notification` получателям и публиковать
   `group_send({type:"notification", data:{…}})` в нужные группы.
5. Добавить `unread_count` в ответ `GET /main/notifications/` и эндпоинт
   `POST /main/notifications/{id}/read/`.
6. Channel layer на Redis для горизонтального масштабирования.
7. Дедупликация: уникальный `id` уведомления (клиент дедупит по нему).

## 8. Связь с фронтендом (что уже сделано)

| Возможность | Реализация на фронте |
|-------------|----------------------|
| WS-подключение | `src/hooks/useNotificationsSocket.js` (`/ws/notifications/?token=`) |
| Авторизация WS | `accessToken` из `localStorage`; refresh при коде 4401 |
| Переподключение | экспоненциальный backoff, reconnect на `online`/возврат вкладки |
| Ping/pong | `{action:"ping"}` каждые 25с |
| Дедупликация | по `id` в `notificationSlice` (`notificationReceived`) |
| Колокольчик | `Header.jsx` — счётчик `unreadCount` из слайса (real-time) |
| Список/история | `NotificationModal` — список из слайса, lazy load по скроллу |
| Прочитать одно | `markNotificationReadAsync` (оптимистично + откат при ошибке) |
| Прочитать все | `markAllNotificationsReadAsync` (оптимистично) |
| Без лишних запросов | первая страница грузится один раз; далее — данные из WS |
| Звук | `src/config/notificationSound.js` → `public/sounds/notification.mp3` (best-effort, заменяемый) |
| Уровень важности | цвет полоски/точки по `level` |
| Переход | клик по уведомлению → `url` (internal `navigate` / external) |

Кроссбраузерность: нативный `WebSocket` + `Audio` (Chrome/Safari/Edge/Firefox),
адаптивный список (десктоп/планшет/мобайл). Звук — best-effort (автоплей может
быть заблокирован до первого взаимодействия — это не ошибка).
