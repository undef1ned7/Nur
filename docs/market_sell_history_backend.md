# Маркет: история продаж (`/crm/sell`) — API для бэкенда

Страница **Продажа** (`/crm/sell`) загружает список через `GET /api/main/pos/sales/` 

**Базовый префикс:** `/api/main/`

---

## `GET /api/main/pos/sales/`

### Query-параметры (использует фронт)

| Параметр | Описание |
|----------|----------|
| `page` | Номер страницы |
| `limit` | Размер страницы (фронт передаёт `100`) |
| `search` | Поиск |
| `user` | UUID продавца (маркет, ограничение по сотруднику) |
| `date_from` | Дата от (`YYYY-MM-DD`, маркет) |
| `date_to` | Дата до (`YYYY-MM-DD`, маркет) |
| `client` | UUID клиента (маркет) |
| `status` | Статус продажи (маркет) |

### Ответ (DRF pagination)

```json
{
  "count": 240,
  "next": "…",
  "previous": null,
  "results": [ … ],
  "meta": {
    "total_amount": "458900.00"
  }
}
```

### `meta.total_amount` — **НОВОЕ**

| Поле | Описание |
|------|----------|
| `meta.total_amount` | Сумма **всех** продаж по текущим фильтрам и поиску, **без** учёта `page` / `limit`. |

**Правила:**

1. Учитываются те же фильтры, что и для списка (`search`, `user`, `date_from`, `date_to`, `client`, `status` и т.д.).
2. Сумма по одной продаже — поле `total` документа (итог чека/продажи).
3. Не суммировать только `results` текущей страницы.
4. Исключённые на фронте отменённые (`canceled`) — по возможности не включать в агрегат на бэкенде или согласовать с фильтром `status`.
