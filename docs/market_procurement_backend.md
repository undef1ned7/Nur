# Маркет: закупки (приёмка от поставщиков) — API для бэкенда

Документация по эндпоинтам, которые использует фронтенд на страницах:

- `/crm/market/procurement` — список закупок (оприходований);
- `/crm/market/procurement/receipt` — приём товара от поставщика.

**Базовый префикс API:** `/api/main/`  
**Авторизация:** как у остальных эндпоинтов `main` (`Authorization: Bearer …`).

---

## 1. Список закупок

### `GET /api/main/suppliers/receipts/`

Список документов приёмки от поставщиков (накладных закупки).

#### Query-параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `page` | int | Номер страницы (с 1). По умолчанию `1`. |
| `limit` | int | Размер страницы. **Фронтенд передаёт `20`.** |
| `supplier_id` | UUID | Фильтр по поставщику |
| `date_from` | date (`YYYY-MM-DD`) | Дата создания документа **от** (включительно) |
| `date_to` | date (`YYYY-MM-DD`) | Дата создания документа **до** (включительно) |

Фильтры комбинируются (логика **И**).

#### Формат ответа (DRF pagination)

```json
{
  "count": 156,
  "next": "https://…/api/main/suppliers/receipts/?page=2&limit=20&…",
  "previous": null,
  "results": [ … ],
  "meta": {
    "total_amount": "1254300.50"
  }
}
```

#### Поле `meta.total_amount` — **НОВОЕ, обязательно для фронта**

| Поле | Тип | Описание |
|------|-----|----------|
| `meta.total_amount` | string (decimal) или number | **Общая сумма всех закупок**, попадающих под текущие фильтры, **без учёта пагинации**. |

**Правила расчёта:**

1. Сумма считается по **всем** документам, которые вернул бы список **без** `page` / `limit` (только с теми же фильтрами `supplier_id`, `date_from`, `date_to`).
2. Сумма **одной закупки** = Σ (`qty` × `purchase_price`) по всем позициям (`items` / `lines`) документа.
3. Если фильтры не заданы — сумма по **всем** закупкам компании (в рамках прав доступа пользователя).
4. Если заданы фильтры — сумма только по **отфильтрованным** закупкам.
5. **Нельзя** возвращать сумму только по `results` текущей страницы — это ошибка для UI «Общая сумма».

Рекомендуемая реализация на бэкенде: один агрегирующий запрос (`Sum(F('qty') * F('purchase_price'))`) с тем же `filter_queryset`, что и для списка, без `LIMIT/OFFSET`.

#### Элемент `results[]`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Идентификатор накладной |
| `supplier_name` | string | Название поставщика |
| `created_by_name` | string \| null | Кто создал |
| `created_at` | datetime (ISO 8601) | Дата создания |
| `items` | array | Позиции (опционально в списке; если пусто — фронт догружает деталь) |
| `total_amount` | string \| number | **Желательно:** сумма накладной на бэкенде (чтобы не считать на клиенте) |

Позиция в `items`:

| Поле | Тип |
|------|-----|
| `product` | UUID или объект товара |
| `qty` | decimal |
| `purchase_price` | decimal |

---

## 2. Деталь закупки

### `GET /api/main/suppliers/receipts/{id}/`

Полная накладная с позициями.

Ответ: объект накладной + `items` (или `lines` — фронт поддерживает оба ключа).

---

## 3. Создание приёмки (приём товара)

### `POST /api/main/suppliers/{supplier_id}/receipt/`

Создание документа оприходования от выбранного поставщика.

**Тело:**

```json
{
  "items": [
    {
      "product": "uuid-товара",
      "qty": "10",
      "purchase_price": "150.00"
    }
  ]
}
```
