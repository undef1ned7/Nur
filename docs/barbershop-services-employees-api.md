# Barbershop Public Services + Employees API (мастера по услугам)

Спецификация изменения публичного эндпоинта услуг и контракта multi-master бронирования для
страницы `/service/{slug}/booking`.

- Базовый URL: `https://app.nurcrm.kg/api`
- Публичные эндпоинты (без авторизации), идентификация компании по `{slug}`.

---

## 1. `GET /api/barbershop/public/{slug}/services/` — услуги + мастера одним запросом

Сейчас эндпоинт возвращает только услуги; мастера грузятся отдельно. Нужно вернуть связи
**услуга → сотрудники** в одном ответе, в **нормализованном** виде (без дублирования сотрудников).

### Рекомендуемая структура (предпочтительно)
```jsonc
{
  "employees": {
    "1": { "id": 1, "name": "Алексей", "avatar": null },
    "2": { "id": 2, "name": "Бек", "avatar": "https://…" }
  },
  "services": [
    { "id": 10, "name": "Стрижка", "category": 3, "category_name": "Волосы",
      "price": "500.00", "time": "30", "employeeIds": [1, 2] },
    { "id": 20, "name": "Бритьё", "category": 3, "category_name": "Волосы",
      "price": "300.00", "time": "20", "employeeIds": [2] }
  ]
}
```

Почему так:
- нет дублирования сотрудников (каждый описан один раз в `employees`);
- меньше размер JSON, быстрее парсинг, проще кеширование;
- одного запроса достаточно для шага выбора мастеров.

### Обратная совместимость
Если бэкенд ещё не обновлён и возвращает **массив** услуг (старый формат) — фронт продолжает
работать: грузит мастеров отдельным запросом `/masters/` и показывает всех (без фильтрации по
услугам). Поэтому переход можно выкатывать поэтапно.

> Допустимый альтернативный формат: вложенный `employees: [{id,name}]` внутри каждой услуги.
> Фронт это тоже понимает (см. парсер в `OnlineBooking.jsx`), но нормализованный с `employeeIds`
> предпочтительнее по размеру/производительности.

### Производительность
- Один запрос; связи service↔employee через `prefetch_related` (M2M), без N+1.
- Индекс на M2M-таблице `service_employees(service_id, employee_id)`.
- Отдавать только активных сотрудников и активные услуги; минимальный набор полей сотрудника.

## 2. `POST /api/barbershop/public/{slug}/bookings/` — бронирование (в т.ч. multi-master)

Шаг «Мастер» позволяет распределить выбранные услуги между несколькими мастерами
(разделённая бронь). Бэкенд должен принимать массив назначений и создавать записи **атомарно**.

### Request (multi-master)
```jsonc
{
  "client_name": "Иван",
  "client_phone": "+996700000000",
  "client_comment": null,
  "date": "2026-06-26",
  "assignments": [
    {
      "master_id": 1,
      "services": [
        { "service_id": 10, "title": "Стрижка", "price": 500, "duration_min": 30 }
      ],
      "time_start": "10:00:00",
      "time_end": "10:30:00"
    },
    {
      "master_id": 2,
      "services": [
        { "service_id": 20, "title": "Бритьё", "price": 300, "duration_min": 20 }
      ],
      "time_start": "10:30:00",
      "time_end": "10:50:00"
    }
  ]
}
```

### Серверная логика
- Транзакция: либо все записи созданы, либо ни одной.
- Валидация: мастер реально умеет назначенные услуги (`employeeIds`); слот свободен; время в
  пределах рабочего дня.
- **Response 201:**
  ```json
  { "bookings": [ { "id": "…", "master_id": 1 }, { "id": "…", "master_id": 2 } ] }
  ```

### Обратная совместимость (fallback фронта)
Если атомарный формат с `assignments` не поддерживается, фронт отправляет **по одному POST на
мастера** в текущем формате (`{ services, master_id, master_name, date, time_start, time_end,
client_* }`) и агрегирует результат/ошибки. Существующая одиночная бронь (один мастер на все
услуги) не меняется.

## Frontend (реализация)

- Парсинг нового формата услуг и построение `employeesById` + `service.employeeIds` —
  `OnlineBooking.jsx` (`load()`), с fallback на старый формат.
- Группировка мастеров по покрытию выбранных услуг и multi-master назначения —
  `src/Components/Sectors/Barber/OnlineBooking/masterGrouping.js` (чистые функции) + UI шага
  «Мастер». Вся группировка — на фронте; бэкенд только отдаёт связи.
