# Маркет — Весы: фильтр весовых товаров в каталоге

**Страница:** `/crm/scales` (фронт: `src/Components/Deposits/Sklad/ScalesPage.jsx`)
**Эндпоинты:** `GET /main/products/list/`, `POST /users/scales/send-products/`
**Статус:** ✅ Фронт работает и без правок бэкенда (фильтрует на клиенте), но нужен серверный фильтр, чтобы не перебирать весь каталог постранично.

---

## 1. Контекст

Страница «Весы» отправляет на торговые весы **только весовые товары**. Весовым считается товар, у которого:

- `is_weight = true`, **или**
- `scale_type = "weight"`.

Сейчас `GET /main/products/list/` не поддерживает фильтр по этим полям, поэтому фронт вынужден:

1. Загружать каталог постранично (`?page=1..N`, до 100 страниц по 100 товаров), пока `next` не станет `null`.
2. Фильтровать весовые на клиенте.
3. Отправлять их на весы явным списком: `POST /users/scales/send-products/` с `{ plu_start: 1, product_ids: [...] }`.

При большом каталоге это десятки лишних запросов ради нескольких весовых позиций.

## 2. Что нужно от бэкенда

### 2.1. Фильтр весовых в списке товаров (обязательно)

```
GET /main/products/list/?is_weight=true
```

| Параметр | Тип | Описание |
|---|---|---|
| `is_weight` | bool (`true`/`false`) | Вернуть только товары, где `is_weight = true` **или** `scale_type = "weight"` (при `false` — все остальные) |

Правила:

1. Фильтр комбинируется с существующими параметрами (`page`, `search` и т.д.).
2. `count` в ответе — количество товаров **после** фильтра (фронт показывает его как «Весовых товаров для выгрузки»).
3. Отсутствие параметра — прежнее поведение (весь каталог), ничего не ломаем.
4. Невалидное значение (`is_weight=abc`) — HTTP 400 с `detail`, а не молчаливое игнорирование.

### 2.2. Отправка на весы (проверить, правок скорее всего не нужно)

`POST /users/scales/send-products/` уже принимает `product_ids` (так склад отправляет выбранные товары). Нужно только подтвердить:

1. При переданном `product_ids` на весы уходят **только** эти товары — никакого «фолбэка на все товары», если какой-то ID не найден.
2. Пустой `product_ids: []` — HTTP 400 с `detail` («нет товаров для отправки»), а не выгрузка всего каталога.

## 3. Поведение фронта после внедрения

Фронт заменит постраничный перебор одним запросом `GET /main/products/list/?is_weight=true` (с пагинацией при необходимости); отправка через `product_ids` останется как есть.

## 4. Чек-лист приёмки

Данные: каталог из 5 товаров — 2 весовых (`is_weight=true` или `scale_type="weight"`), 3 штучных.

| # | Запрос | Ожидание |
|---|---|---|
| 1 | `GET /main/products/list/?is_weight=true` | `count=2`, в `results` только весовые |
| 2 | `GET /main/products/list/?is_weight=true&search=<имя весового>` | фильтры комбинируются |
| 3 | `GET /main/products/list/` (без параметра) | `count=5`, поведение не изменилось |
| 4 | `GET /main/products/list/?is_weight=abc` | HTTP 400 с `detail` |
| 5 | `POST /users/scales/send-products/` c `product_ids` из двух весовых | на весы уходят ровно 2 товара |
| 6 | `POST /users/scales/send-products/` c `product_ids: []` | HTTP 400, каталог целиком **не** выгружается |
