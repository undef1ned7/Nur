# Маркет — Чек (receipt JSON): изменение цены ≠ скидка

**Экран:** `/crm/market/cashier` (фронт: `src/Components/Sectors/Market/CashierPage/CashierPage.jsx`)
**Эндпоинты:**
- `PATCH /main/pos/carts/{cart_id}/items/{item_id}/` — изменение строки корзины (цена / скидка / кол-во)
- `GET /main/sales/json/{sale_id}/receipt/` — JSON чека для печати
**Статус:** ⚠️ Требуется правка на бэкенде — ручное изменение цены товара кассиром попадает в поле `discount` чека, из-за чего скидка на печати вычитается второй раз.

---

## 1. Проблема

Кассир в POS Маркета **вручную изменил цену** товара с 250 на 230 (никакой скидки не оформлял). Бэкенд вернул чек, где разница цен записана как скидка:

```json
// Чек № 1322 — цену изменили 250 → 230, скидок НЕТ
{
    "items": [
        { "name": "Лемонад Kt", "qty": 1.0, "price": 230.0,
          "line_discount": 0.0, "line_total": 230.0 }
    ],
    "discount": 20.0,        // ← ОШИБКА: это не скидка, это изменение цены
    "paid_cash": 230.0,
    "change": 0.0
}
```

Система печати (и предпросмотр чека) считает итог по контракту `Итог = Подытог − discount`:

- Подытог (валовый) = Σ(`line_total` + `line_discount`) = 230
- Итог = 230 − 20 = **210.00** — а наличными принято 230, сдача 0.

На чеке покупатель видит итог 210 при фактической оплате 230 — данные чека противоречат сами себе. Сумма 20 уже «зашита» в `price`/`line_total`, и повторное её вычитание через `discount` — двойной учёт.

Для сравнения — корректный чек с настоящей скидкой:

```json
// Чек № 1323 — цена 250, скидка на строку 10, скидка на заказ 10
{
    "items": [
        { "name": "Лемонад Kt", "qty": 1.0, "price": 250.0,
          "line_discount": 10.0, "line_total": 240.0 }
    ],
    "discount": 20.0,        // 10 (строка) + 10 (заказ) — всё сходится
    "paid_cash": 230.0
}
```

Здесь: Подытог = 240 + 10 = 250, Итог = 250 − 20 = **230** = `paid_cash`. ✅

## 2. Как фронт меняет цену и скидку (важно: это независимые операции)

Фронт **никогда не отправляет изменение цены как скидку**. Это два разных PATCH-запроса на `/main/pos/carts/{cart_id}/items/{item_id}/`:

| Действие кассира | Тело запроса | Что это значит |
|---|---|---|
| Изменил цену | `{ "unit_price": "230.00" }` | Товар продаётся по новой цене. Скидки нет. |
| Дал скидку на строку | `{ "discount_total": "10.00" }` | Скидка на позицию в сомах. |
| Дал скидку на заказ | `{ "order_discount_total": "10.00" }` или `{ "order_discount_percent": 5 }` (PATCH продажи) | Скидка на весь чек. |

Код фронта: `patchCartItemPrice` шлёт только `unit_price` (`CashierPage.jsx:2825`), `patchCartItemDiscount` — только `discount_total` (`CashierPage.jsx:2879`).

## 3. Требуемое поведение (обязательные правила)

1. **Изменение `unit_price` не должно порождать скидку.** Разница между базовой (каталожной) ценой товара и `unit_price`, введённой кассиром, — это НЕ скидка. Она не должна попадать ни в `line_discount` строки, ни в общий `discount` чека, ни в фискальные данные eKassa.
2. **`price` в чеке = фактическая цена продажи** (та, что кассир видел/ввёл, т.е. `unit_price` строки корзины). Не каталожная.
3. **`line_discount` = только явная скидка на строку**, оформленная через `discount_total`.
4. **`discount` (на уровне чека) = скидка на заказ + Σ построчных скидок.** Только явно оформленные скидки. Инвариант контракта печати:

   ```
   Σ(line_total + line_discount) − discount + tax = сумма к оплате = Σ payments.amount
   ```

   Этот инвариант обязан выполняться для каждого чека. Чек 1322 его нарушает (230 − 20 ≠ 230), чек 1323 — соблюдает (250 − 20 = 230).
5. **`line_total` = `unit_price` × `qty` − `line_discount`** — и никаких других вычетов.
6. Если хочется хранить отклонение от каталожной цены для внутренней аналитики — храните его в отдельном поле (например, `price_adjustment`), но не в `discount`/`line_discount` печатного чека.

## 4. Приёмочные сценарии

| # | Действия кассира | Ожидаемый чек |
|---|---|---|
| 1 | Товар 250, цену изменили на 230, скидок нет | `price: 230, line_discount: 0, line_total: 230, discount: 0`, итог 230 |
| 2 | Товар 250, скидка на строку 10 | `price: 250, line_discount: 10, line_total: 240, discount: 10`, итог 240 |
| 3 | Товар 250, скидка на строку 10 + скидка на заказ 10 | `price: 250, line_discount: 10, line_total: 240, discount: 20`, итог 230 |
| 4 | Товар 250, цену изменили на 230, потом скидка на строку 10 | `price: 230, line_discount: 10, line_total: 220, discount: 10`, итог 220 |

Во всех сценариях `paid_cash`/`payments` должны сходиться с итогом, а `change` = `cash_received` − итог.

## 5. Где это стреляет на фронте

- **Печать чека** — `buildReceiptFromJSON` (`src/Components/pages/Sell/services/printService.js`): печатает `Подытог − discount`, при лишнем `discount` итог на бумаге меньше реально оплаченного.
- **Предпросмотр чека** — `ReceiptPreviewModal` (Маркет, кассир и документы): та же математика.
- **eKassa** — при фискализации суммы скидок уезжают в поля 1059/1031; лишняя «скидка» даст расхождение фискального итога с оплатой.
